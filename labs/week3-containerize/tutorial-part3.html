11
backend/Dockerfile
Docker

เปิดไฟล์ backend/Dockerfile — ไฟล์นี้บอก Docker วิธี build backend เป็น image

เปิดไฟล์: backend/Dockerfile
backend/Dockerfile — เนื้อหาทั้งหมด
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY . .
EXPOSE 3001
CMD ["node", "index.js"]

อธิบายแต่ละบรรทัด:

FROM node:20-alpinebase image — node.js 20 บน Alpine Linux (เล็กกว่า debian ~3x) ขนาดประมาณ 50MB
WORKDIR /appกำหนด working directory ภายใน container — คำสั่งถัดไปทั้งหมดรันใน /app
COPY package*.json ./copy เฉพาะ package.json + package-lock.json ก่อน — ใช้ประโยชน์ Docker layer cache
RUN npm ci --omit=devติดตั้งเฉพาะ production deps (ไม่รวม nodemon ฯลฯ) ใช้ ci แทน install เพราะล็อค version จาก lock file
COPY . .copy source code ทั้งหมด — ทำทีหลัง package.json เพราะ code เปลี่ยนบ่อยกว่า dependencies
EXPOSE 3001บอก Docker ว่า container ใช้ port 3001 (documentation เท่านั้น — ต้องกำหนดใน compose ด้วย)
CMD ["node", "index.js"]คำสั่งรัน container — ใช้ JSON array form เพื่อให้ process รับ signal ได้ถูกต้อง
⚡
Layer Caching ทำงานยังไง? Docker เก็บ cache แต่ละ instruction เป็น layer
— ถ้า package.json ไม่เปลี่ยน → layer npm ci ถูก cache → build เร็วขึ้นมาก
— ถ้าแก้ code ใน index.js → แค่ layer COPY . . ถูก re-run เท่านั้น
🚫
backend/.dockerignore ป้องกัน copy ไฟล์ที่ไม่ต้องการ ไฟล์ backend/.dockerignore ระบุให้ Docker ข้าม: node_modules/, .env, *.log ทำให้ build เร็วขึ้นและ image ไม่มี local .env file ที่อาจมี secrets
12
frontend/Dockerfile + nginx.conf (Multi-stage)
Docker

Frontend ใช้ Multi-stage build — stage แรก build Vue app, stage สองเอาแค่ผลลัพธ์ไป serve ด้วย Nginx

Single-stage (ไม่ใช้)
~1.2 GB
Node.js + npm + source code + node_modules ทั้งหมด
Multi-stage ✓
~25 MB
แค่ Nginx + static files (HTML/CSS/JS) เท่านั้น
เปิดไฟล์: frontend/Dockerfile
frontend/Dockerfile — เนื้อหาทั้งหมด
# ═══ Stage 1: Build Vue App ══════════════════════════════════════
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build
# ผลลัพธ์จาก npm run build อยู่ที่ /app/dist/

# ═══ Stage 2: Serve with Nginx ═══════════════════════════════════
FROM nginx:alpine
# copy เฉพาะ /app/dist จาก stage 1 — ไม่มี Node.js ใน image สุดท้าย
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
AS builderตั้งชื่อ stage 1 ว่า "builder" เพื่อ reference ในภายหลัง
npm run buildVite compile Vue → static files (HTML, JS, CSS) ลงใน /app/dist/
FROM nginx:alpineเริ่ม stage ใหม่ด้วย Nginx — ทุกอย่างจาก stage 1 (Node.js, source code) หายไป
--from=buildercopy files จาก stage ชื่อ "builder" — ดึงเฉพาะ /app/dist มา
nginx.confconfig บอก Nginx ว่าจะ serve files ยังไง และ proxy /api ไปที่ backend
daemon offรัน Nginx แบบ foreground — Docker ต้องการ process หลักอยู่ใน foreground
เปิดไฟล์: frontend/nginx.conf
frontend/nginx.conf — เนื้อหาทั้งหมด
server {
    listen 80;
    root /usr/share/nginx/html;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api {
        proxy_pass         http://backend:3001;
        proxy_http_version 1.1;
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
    }
}
try_files ... /index.htmlถ้าไม่เจอไฟล์ตาม path → ส่ง index.html — ทำให้ Vue Router ทำงานได้ถ้ามีการใช้
proxy_pass http://backend:3001"backend" = ชื่อ service ใน docker-compose.yml — Docker DNS resolve เป็น IP container อัตโนมัติ
location /apirequest ที่ URL ขึ้นต้นด้วย /api → forward ไป backend แทนที่จะ serve static file
🔀
vite.config.js vs nginx.conf — Proxy อยู่ที่ไหน? Dev mode: Vite dev server ทำหน้าที่ proxy (vite.config.js)
Production (Docker): ไม่มี Vite — Nginx ทำหน้าที่ proxy (nginx.conf)
ทั้งคู่ทำอย่างเดียวกัน: ส่ง /api/* ไป backend:3001
13
docker-compose.yml — รัน 3 Services พร้อมกัน
Compose

เปิดไฟล์ docker-compose.yml — ไฟล์นี้กำหนด 3 services: db, backend, frontend

เปิดไฟล์: docker-compose.yml
docker-compose.yml — เนื้อหาทั้งหมด
services:

  # ── Database ─────────────────────────────────────────────────
  db:
    image: postgres:15-alpine
    restart: unless-stopped
    environment:
      POSTGRES_DB:       stockpro
      POSTGRES_USER:     dev
      POSTGRES_PASSWORD: secret
    volumes:
      - pg_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U dev -d stockpro"]
      interval: 5s
      timeout: 5s
      retries: 5

  # ── Backend ───────────────────────────────────────────────────
  backend:
    build: ./backend
    restart: unless-stopped
    ports:
      - "3001:3001"
    environment:
      DB_HOST:     db
      DB_PORT:     5432
      DB_NAME:     stockpro
      DB_USER:     dev
      DB_PASSWORD: secret
      PORT:        3001
    depends_on:
      db:
        condition: service_healthy

  # ── Frontend ──────────────────────────────────────────────────
  frontend:
    build: ./frontend
    restart: unless-stopped
    ports:
      - "5173:80"
    depends_on:
      - backend

volumes:
  pg_data:

อธิบาย key concepts:

image: postgres:15-alpineใช้ official PostgreSQL image จาก Docker Hub โดยตรง (ไม่ต้อง build)
volumes: pg_data:/var/lib/postgresql/datanamed volume เก็บข้อมูลไว้ถาวร — restart หรือ recreate container ข้อมูลไม่หาย
healthcheckตรวจสอบว่า postgres พร้อมรับ connection แล้ว (pg_isready) — ทุก 5 วินาที, retry 5 ครั้ง
build: ./backendbuild image จาก Dockerfile ใน ./backend — ทำครั้งแรกหรือเมื่อ --build
DB_HOST: db"db" = ชื่อ service — Docker สร้าง DNS ให้อัตโนมัติ backend คุยกับ db ผ่านชื่อนี้
condition: service_healthyรอให้ db healthy ก่อน start backend — ไม่งั้น backend อาจ start แล้ว db ยังไม่พร้อม
ports: "5173:80"host:container — Nginx รันที่ port 80 ใน container, เข้าถึงได้ที่ localhost:5173 บน host
🕸️
Docker Network อัตโนมัติ Docker Compose สร้าง private network ให้ทุก service อัตโนมัติ services คุยกันได้ผ่านชื่อ service โดยไม่ต้อง expose port ถึงกัน เฉพาะ ports ที่ระบุเท่านั้นที่ expose ออก host

Build & ทดสอบ Full Stack ด้วย Docker Compose
Test
⚠️
หยุด local backend และ frontend dev server ก่อน Ctrl+C ใน terminal ที่รัน node index.js และ npm run dev เพราะจะใช้ port เดิม (3001 และ 5173) ซ้ำกัน

Build และรัน 3 services พร้อมกัน:

terminal — อยู่ใน week3-containerize/
# Build images + รัน (แสดง logs ใน terminal)
docker compose up --build

# หรือ รัน background mode
docker compose up --build -d

ดู output ที่ควรเห็น (ลำดับการ start):

ตัวอย่าง logs จาก docker compose up --build
[+] Building ...

 ✓ frontend Built in xx.xs

[+] Running 3/3
 ✓ Container week3-db-1        Healthy    ← db พร้อมก่อน
 ✓ Container week3-backend-1   Started    ← backend รอ db
 ✓ Container week3-frontend-1  Started    ← frontend รอ backend

week3-backend-1  | ✅ DB connected at: 2025-xx-xx...
week3-backend-1  | ✅ Table products ready
week3-backend-1  | 🌱 Seed data inserted (10 products)
week3-backend-1  | 🚀 StockPro API running at http://localhost:3001

คำสั่ง debug ที่มีประโยชน์:

terminal — debug commands
# ดู status ของทุก service
docker compose ps

# ดู logs real-time (กด Ctrl+C เพื่อหยุด)
docker compose logs -f

# ดู logs เฉพาะ backend
docker compose logs -f backend

# เข้าไปใน container เพื่อ debug
docker compose exec backend sh
docker compose exec db psql -U dev -d stockpro

# หยุดและลบ containers (ข้อมูลใน pg_data volume ยังอยู่)
docker compose down

# หยุดและลบทั้ง containers + volumes (ข้อมูลหายด้วย)
docker compose down -v
✅
ตรวจสอบ Full Stack
  • docker compose ps → ทุก service status เป็น "running" หรือ "healthy"
  • เปิด http://localhost:5173 → เห็น StockPro UI พร้อม products 10 รายการ
  • เพิ่มสินค้าใหม่ → refresh หน้า → สินค้ายังอยู่ (persisted ใน volume)
  • รัน docker compose down แล้ว docker compose up -d ใหม่ → ข้อมูลยังอยู่
  • http://localhost:3001/health → {"status":"ok",...}

Push Images สู่ Docker Hub + ส่งงาน
Docker Hub

Push Docker images ขึ้น Docker Hub เพื่อให้ classmate หรือ server ใดก็ pull image ได้ทันทีโดยไม่ต้อง build เอง

📋
ต้องมีก่อน: บัญชี Docker Hub (ฟรี) ที่ hub.docker.com และจำ Username ไว้ — ชื่อ image จะเป็น USERNAME/stockpro-backend:latest

ขั้นตอนที่ 1 — Login Docker Hub:

terminal
docker login
# Username: your-dockerhub-username
# Password: ********  ← ใส่ password หรือ Access Token
# → Login Succeeded
🔐
แนะนำ: ใช้ Access Token แทน Password hub.docker.com → Account Settings → Personal Access Tokens → New Access Token ปลอดภัยกว่า และ revoke ได้ทีหลัง

ขั้นตอนที่ 2 — Build images พร้อม Tag ชื่อ Docker Hub:

terminal — แทน YOUR_USERNAME ด้วย Docker Hub username ของตัวเอง
# Bash / Git Bash
export HUB_USER=YOUR_USERNAME

# Windows PowerShell
$env:HUB_USER="YOUR_USERNAME"

# Build backend image
docker build -t $HUB_USER/stockpro-backend:latest ./backend

# Build frontend image
docker build -t $HUB_USER/stockpro-frontend:latest ./frontend

# ตรวจสอบ images ที่ได้
docker images | grep stockpro

ขั้นตอนที่ 3 — Push ขึ้น Docker Hub:

terminal
docker push $HUB_USER/stockpro-backend:latest
docker push $HUB_USER/stockpro-frontend:latest

# ตรวจสอบ: เปิด browser ไปที่
# https://hub.docker.com/r/YOUR_USERNAME/stockpro-backend

ขั้นตอนที่ 4 — ทดสอบ pull และรัน Production (ไม่ต้องมี source code):

เปิดไฟล์ docker-compose.prod.yml แล้วแก้ YOUR_USERNAME เป็น username จริง:

แก้ไขไฟล์: docker-compose.prod.yml
docker-compose.prod.yml — ตัวอย่าง (แก้ YOUR_USERNAME)
services:

  db:
    image: postgres:15-alpine
    restart: unless-stopped
    environment:
      POSTGRES_DB:       stockpro
      POSTGRES_USER:     dev
      POSTGRES_PASSWORD: secret
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U dev -d stockpro"]
      interval: 5s
      timeout: 5s
      retries: 5

  backend:
    image: YOUR_USERNAME/stockpro-backend:latest   # ← แก้ตรงนี้
    restart: unless-stopped
    ports:
      - "3001:3001"
    environment:
      DB_HOST: db
      DB_PORT: 5432
      DB_NAME: stockpro
      DB_USER: dev
      DB_PASSWORD: secret
      PORT: 3001
    depends_on:
      db:
        condition: service_healthy

  frontend:
    image: YOUR_USERNAME/stockpro-frontend:latest  # ← แก้ตรงนี้
    restart: unless-stopped
    ports:
      - "5173:80"
    depends_on:
      - backend

volumes:
  pgdata:
terminal — รัน production (pull จาก Docker Hub)
# Pull images จาก Docker Hub
docker compose -f docker-compose.prod.yml pull

# รัน
docker compose -f docker-compose.prod.yml up -d

# ทดสอบ: เปิด http://localhost:5173
✅
ตรวจสอบ Docker Hub
  • เข้า hub.docker.com/r/YOUR_USERNAME/stockpro-backend → เห็น repository + tag "latest"
  • เข้า hub.docker.com/r/YOUR_USERNAME/stockpro-frontend → เห็น repository เหมือนกัน
  • รัน docker-compose.prod.yml บนเครื่องอื่น → StockPro ทำงานได้โดยไม่มี source code
