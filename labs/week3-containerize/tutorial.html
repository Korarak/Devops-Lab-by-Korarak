Database Schema — ตาราง products
Database

ต่างจาก Todo ที่มีแค่ title กับ completed — ตาราง products มีฟิลด์ที่สื่อความหมายของสินค้าจริง

SQL — สร้างตาราง products
CREATE TABLE IF NOT EXISTS products (
  id          SERIAL PRIMARY KEY,          -- รหัสอัตโนมัติ
  name        VARCHAR(150) NOT NULL,       -- ชื่อสินค้า
  category    VARCHAR(80)  NOT NULL,       -- หมวดหมู่
  price       NUMERIC(12,2) NOT NULL DEFAULT 0,   -- ราคา
  stock       INTEGER      NOT NULL DEFAULT 0,    -- จำนวนสต็อก
  description TEXT         DEFAULT '',    -- คำอธิบาย (ถ้ามี)
  created_at  TIMESTAMPTZ  DEFAULT NOW(),  -- วันที่เพิ่ม
  updated_at  TIMESTAMPTZ  DEFAULT NOW()   -- วันที่แก้ไขล่าสุด
);
💡
ไม่ต้องรัน SQL นี้เอง Backend จะ auto-create ตารางนี้ตอน startup ผ่าน initDb() function และยังมี Seed Data 10 รายการให้ลองใช้งานทันที

ความแตกต่างของ Data Type:

  • NUMERIC(12,2) — ราคา รองรับทศนิยม 2 ตำแหน่ง (ใช้แทน FLOAT เพราะแม่นยำกว่าสำหรับเงิน)
  • INTEGER — จำนวนสต็อก เป็นตัวเลขจำนวนเต็มเสมอ
  • TIMESTAMPTZ — Timestamp พร้อม Timezone (ดีกว่า TIMESTAMP เพราะรองรับ UTC)

Backend API — Products CRUD
Express

Backend ของ StockPro มี endpoints มากกว่า Todo — เพิ่ม GET categories และ Search/Filter ผ่าน query parameters

รายการ API Endpoints ทั้งหมด:

Method URL ทำอะไร
GET/api/products?search=&category=ดึงรายการสินค้า + filter
GET/api/categoriesดึงรายการ category ทั้งหมด
POST/api/productsเพิ่มสินค้าใหม่
PUT/api/products/:idแก้ไขสินค้า
DELETE/api/products/:idลบสินค้า

ส่วนที่น่าสนใจใน backend/index.js:

backend/index.js — GET with Search + Filter
app.get('/api/products', async (req, res) => {
  const { search = '', category = '' } = req.query;
  const params = [];
  let where = 'WHERE 1=1';              // เริ่มต้นด้วย condition ที่เป็น true เสมอ

  if (search) {
    params.push(`%${search}%`);         // % คือ wildcard ใน SQL LIKE
    where += ` AND (name ILIKE $${params.length} OR description ILIKE $${params.length})`;
    // ILIKE = case-insensitive LIKE (ค้นได้ทั้ง uppercase/lowercase)
  }
  if (category) {
    params.push(category);
    where += ` AND category = $${params.length}`;
  }

  const { rows } = await pool.query(
    `SELECT * FROM products ${where} ORDER BY created_at DESC`, params
  );
  res.json(rows);
});
🔒
ทำไมต้องใช้ Parameterized Query ($1, $2)? ป้องกัน SQL Injection — ถ้า user พิมพ์ '; DROP TABLE products; -- ในช่อง search Parameterized Query จะส่งเป็น string ธรรมดา ไม่ execute เป็น SQL

Vue 3 vs Vanilla JS — เปรียบเทียบชัดๆ
Vue 3

ก่อนเขียน Vue 3 ให้เข้าใจก่อนว่า Vue 3 แก้ปัญหาอะไรของ Vanilla JS — ลองดูตัวอย่างการดึงและแสดงสินค้า

เปรียบเทียบ: ดึงสินค้าและแสดงบนหน้าเว็บ
// ❌ Vanilla JS — ต้อง manipulate DOM เอง async function fetchProducts() { const res = await fetch('/api/products') const products = await res.json() const list = document.getElementById('list') list.innerHTML = products.map(p => `<div>${p.name} — ฿${p.price}</div>` ).join('') // ต้องทำใหม่ทุกครั้งที่ข้อมูลเปลี่ยน! } // ✓ Vue 3 — แค่เปลี่ยน data, Vue อัพเดต DOM ให้อัตโนมัติ const products = ref([]) // reactive state async function fetchProducts() { const res = await fetch('/api/products') products.value = await res.json() // Vue detect การเปลี่ยนแปลง // DOM อัพเดตอัตโนมัติ ไม่ต้อง innerHTML ! } // ใน template — Vue render loop ให้อัตโนมัติ // <div v-for="p in products" :key="p.id"> // {{ p.name }} — ฿{{ p.price }} // </div>
⚡
Reactivity คืออะไร? Vue 3 ติดตาม ref() และ computed() อัตโนมัติ — เมื่อ products.value เปลี่ยน Vue จะ re-render เฉพาะส่วนที่ใช้ products เท่านั้น (Virtual DOM diffing) ทำให้เร็วและมีประสิทธิภาพ

import { defineConfig } from 'vite'

export default defineConfig({
  plugins: [vue()],
  server: {
    proxy: {
      // เมื่อ dev server เห็น /api → ส่งต่อไปที่ backend
      '/api': 'http://localhost:3001'
    }
  }
})

Stats Dashboard — 4 Metric Cards
Vue 3

Dashboard แสดงข้อมูลสรุปของสินค้าทั้งหมด — ค่าเหล่านี้คำนวณจากข้อมูลด้วย computed() ไม่ต้องเรียก API เพิ่ม

App.vue — computed stats
// computed() จะ recalculate อัตโนมัติเมื่อ products.value เปลี่ยน
const stats = computed(() => ({
  total:      products.value.length,
  // สินค้าที่สต็อกน้อยกว่า 10 ชิ้น
  lowStock:   products.value.filter(p => p.stock < 10).length,
  // รวมจำนวนสต็อกทั้งหมด
  totalItems: products.value.reduce((sum, p) => sum + p.stock, 0),
  // มูลค่ารวม = ราคา × จำนวน
  totalValue: products.value.reduce(
    (sum, p) => sum + parseFloat(p.price) * p.stock, 0
  )
}))
App.vue template — Stats Grid
<div class="stats-grid">
  <div class="stat-card">
    <div class="stat-icon si-green">📦</div>
    <div>
      <div class="stat-val">{{ stats.total }}</div>
      <div class="stat-label">สินค้าทั้งหมด</div>
    </div>
  </div>
  <!-- ... อีก 3 cards -->
</div>

<!-- แสดง alert banner ถ้ามีสต็อกใกล้หมด -->
<div class="alert-low" v-if="stats.lowStock > 0">
  🚨 มีสินค้า <strong>{{ stats.lowStock }} รายการ</strong>
  ที่มีสต็อกน้อยกว่า 10 ชิ้น
</div>
🧮
computed() vs methods() computed() มี cache — ถ้า products.value ไม่เปลี่ยน Vue จะใช้ค่าที่คำนวณไว้แล้ว ไม่ต้อง recalculate ใหม่ ต่างจาก methods() ที่รันทุกครั้งที่ template re-render ทำให้ computed() เหมาะกับการคำนวณที่ใช้ทรัพยากร

Product Grid & Cards
Vue 3

Product card แต่ละใบแสดง: ชื่อ, category badge (สีต่างกันตาม category), ราคา, และ stock level bar ที่เปลี่ยนสีตามจำนวน

App.vue — Product Grid template
<div class="product-grid">
  <div
    v-for="p in filtered"     <!-- loop ผ่าน filtered (ไม่ใช่ products ดิบ) -->
    :key="p.id"               <!-- key ช่วย Vue track แต่ละ element -->
    class="product-card"
    :class="{ 'card-low': p.stock > 0 && p.stock < 10, 'card-out': p.stock <= 0 }"
  >
    <!-- Category Badge สีต่างกันตาม category -->
    <span class="cat-badge" :class="catClass(p.category)">{{ p.category }}</span>

    <!-- Stock Level Bar -->
    <div class="stock-track">
      <div
        class="stock-fill"
        :class="stockClass(p.stock)"
        :style="{ width: Math.min((p.stock / 80) * 100, 100) + '%' }"
      ></div>
    </div>

    <!-- Action Buttons -->
    <button @click="openEdit(p)">✏️ แก้ไข</button>
    <button @click="confirmDelete = p">🗑️ ลบ</button>
  </div>
</div>
App.vue script — stockClass() helper
// กำหนด CSS class ตามจำนวนสต็อก — ใช้กับ stock bar และตัวเลข
function stockClass(s) {
  if (s <= 0)  return 'out'   // หมดแล้ว → สีเทา
  if (s < 10)  return 'low'   // น้อยกว่า 10 → สีแดง
  if (s < 30)  return 'mid'   // 10-29 → สีเหลือง
  return 'high'               // 30+ → สีเขียว
}

Search & Filter ด้วย Computed
Vue 3

Search และ Filter ทำงานด้วย computed() — ผลลัพธ์จะ update ทันทีเมื่อ search หรือ catFilter เปลี่ยน โดยไม่ต้องเรียก API ใหม่

App.vue script — search + filter logic
const search    = ref('')    // ข้อความค้นหา
const catFilter = ref('')    // category ที่เลือก
let   debounceTimer = null   // สำหรับ debounce

// computed ที่กรองสินค้าตาม search + category
const filtered = computed(() => {
  const s = search.value.toLowerCase()
  return products.value.filter(p => {
    // ตรงกับ search? (case-insensitive)
    const matchSearch = !s ||
      p.name.toLowerCase().includes(s) ||
      (p.description || '').toLowerCase().includes(s)
    // ตรงกับ category filter?
    const matchCat = !catFilter.value || p.category === catFilter.value
    return matchSearch && matchCat
  })
})

// debounce: รอ 280ms หลัง user หยุดพิมพ์แล้วค่อย fetch
// (ป้องกัน API ถูกเรียกทุก keystroke)
function debounceFetch() {
  clearTimeout(debounceTimer)
  debounceTimer = setTimeout(fetchProducts, 280)
}
App.vue template — Toolbar
<div class="toolbar">
  <!-- v-model ผูก input กับ search ref -->
  <!-- @input เรียก debounceFetch แทน fetchProducts โดยตรง -->
  <input v-model="search" @input="debounceFetch" placeholder="🔍 ค้นหา..." />

  <!-- v-model ผูก select กับ catFilter ref -->
  <select v-model="catFilter" @change="fetchProducts">
    <option value="">ทุกหมวดหมู่</option>
    <!-- loop ผ่าน categories computed -->
    <option v-for="c in categories" :key="c" :value="c">{{ c }}</option>
  </select>

  <span>แสดง {{ filtered.length }} / {{ products.length }} รายการ</span>
</div>

Modal Form (Add/Edit) + Delete Confirm
Vue 3

Modal แสดง/ซ่อนด้วย v-if — ข้อมูลในฟอร์มผูกกับ form ref ด้วย v-model

App.vue script — Modal logic
const showModal    = ref(false)       // แสดง/ซ่อน modal
const editingId    = ref(null)        // null = กำลัง add, มีค่า = กำลัง edit
const confirmDelete = ref(null)       // เก็บ product ที่กำลังจะลบ

// form object ที่ผูกกับ input fields ใน modal
const form = ref({ name: '', category: '', price: '', stock: 0, description: '' })

function openAdd() {
  editingId.value = null                    // clear editing state
  form.value = { name: '', category: '', price: '', stock: 0, description: '' }
  showModal.value = true
}

function openEdit(p) {
  editingId.value = p.id                    // บันทึกว่ากำลัง edit id ไหน
  form.value = {                            // copy ค่าจาก product เข้า form
    name: p.name, category: p.category,
    price: p.price, stock: p.stock, description: p.description || ''
  }
  showModal.value = true
}

async function saveProduct() {
  const body = {
    name: form.value.name, category: form.value.category,
    price: parseFloat(form.value.price),    // convert string → number
    stock: parseInt(form.value.stock),
    description: form.value.description
  }
  // ใช้ URL และ method ต่างกันตาม Add หรือ Edit
  const url    = editingId.value ? `/api/products/${editingId.value}` : '/api/products'
  const method = editingId.value ? 'PUT' : 'POST'
  await fetch(url, { method, headers: { 'Content-Type': 'application/json' }, body: JSON.stringify(body) })
  showModal.value = false
  fetchProducts()    // reload สินค้าใหม่
}
App.vue template — Modal (ตัดสั้น)
<!-- v-if แสดง modal เมื่อ showModal เป็น true -->
<!-- @click.self = คลิก overlay (ไม่ใช่ modal) ถึงจะปิด -->
<div class="overlay" v-if="showModal" @click.self="showModal = false">
  <div class="modal">
    <h2>{{ editingId ? 'แก้ไขสินค้า' : 'เพิ่มสินค้าใหม่' }}</h2>
    <form @submit.prevent="saveProduct">
      <!-- v-model ผูก input กับ form.name อัตโนมัติ -->
      <input v-model="form.name" required />
      <!-- datalist สร้าง autocomplete จาก categories ที่มีอยู่ -->
      <input v-model="form.category" list="cat-list" />
      <datalist id="cat-list">
        <option v-for="c in categories" :value="c" />
      </datalist>
      <button type="submit">{{ editingId ? 'บันทึก' : 'เพิ่ม' }}</button>
    </form>
  </div>
</div>

RUN npm ci --only=production

# Copy source code ทีหลัง (เปลี่ยนบ่อย → layer บน = invalidate น้อยกว่า)
COPY . .

EXPOSE 3001

HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -qO- http://localhost:3001/health || exit 1

CMD ["node", "index.js"]

frontend/Dockerfile (Multi-stage):

frontend/Dockerfile
# ═══ Stage 1: Build ════════════════════════════════════════
# ใช้ image ใหญ่ที่มี Node.js เต็มๆ สำหรับ build เท่านั้น
FROM node:20-alpine AS builder

WORKDIR /app
COPY package*.json ./
RUN npm ci

COPY . .
# npm run build สร้างไฟล์ static ใน /app/dist/
RUN npm run build

# ═══ Stage 2: Production ════════════════════════════════════
# ใช้ Nginx image เล็กๆ เพื่อ serve ไฟล์ static เท่านั้น
# ไม่มี Node.js, npm, หรือ source code → image เล็ก + ปลอดภัยกว่า
FROM nginx:alpine

# Copy เฉพาะผลลัพธ์จาก Stage 1 มา
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
📦
ทำไม Multi-stage ถึงสำคัญ? node:20-alpine มีขนาด ~180MB + dependencies อาจรวมถึง 1GB+
nginx:alpine มีขนาดแค่ ~25MB — เหมาะสำหรับ production ที่ต้องการแค่ serve HTML/JS/CSS

Docker Compose — 3 Services รวมกัน
Compose

docker-compose.yml เชื่อม 3 services เข้าด้วยกันในรัน network เดียว — services คุยกันผ่านชื่อ (DNS resolution)

docker-compose.yml
services:
  # ── Database ─────────────────────────────────────────────────
  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_DB:       stockpro
      POSTGRES_USER:     dev
      POSTGRES_PASSWORD: secret
    volumes:
      - pgdata:/var/lib/postgresql/data   # persist data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U dev -d stockpro"]
      interval: 10s
      timeout: 5s
      retries: 5

  # ── Backend ───────────────────────────────────────────────────
  backend:
    build: ./backend
    ports:
      - "3001:3001"
    environment:
      DATABASE_URL: postgresql://dev:secret@db:5432/stockpro
      # db = ชื่อ service ด้านบน — Docker DNS แก้ไขเป็น IP อัตโนมัติ
    depends_on:
      db:
        condition: service_healthy   # รอให้ DB healthy ก่อน start

  # ── Frontend ──────────────────────────────────────────────────
  frontend:
    build: ./frontend
    ports:
      - "5173:80"     # host:container (Nginx รันที่ 80 ใน container)
    depends_on:
      - backend

volumes:
  pgdata:             # Named volume — ข้อมูลไม่หายแม้ restart
💡
ทำไม Frontend ต้อง proxy ไปที่ backend? Nginx ใน container ต้องรู้ว่าจะส่ง /api/* ไปที่ไหน — ดู nginx.conf ซึ่งตั้งค่า proxy_pass http://backend:3001/; (ใช้ชื่อ service แทน IP)

Push Images สู่ Private Registry
Registry

Push images ขึ้น Private Registry เพื่อให้ server อื่น (หรือ classmate) สามารถ pull และรัน StockPro ได้โดยไม่ต้อง build เอง

terminal — tag & push images
# แทน REGISTRY_IP ด้วย IP ของ Private Registry โรงเรียน
export REGISTRY=192.168.x.x:5000

# Login (ถ้า Registry มี authentication)
docker login $REGISTRY

# Build images พร้อม tag สำหรับ registry
docker build -t $REGISTRY/stockpro-backend:latest ./backend
docker build -t $REGISTRY/stockpro-frontend:latest ./frontend

# Push ขึ้น registry
docker push $REGISTRY/stockpro-backend:latest
docker push $REGISTRY/stockpro-frontend:latest

# ยืนยัน: ทดลอง pull กลับมาดู
docker pull $REGISTRY/stockpro-backend:latest
docker-compose.prod.yml — ใช้ image จาก Registry
services:
  backend:
    image: 192.168.x.x:5000/stockpro-backend:latest   # ดึงจาก Registry
    # ไม่มี build: — ใช้ image ที่ build ไว้แล้ว
    ports: ["3001:3001"]
    environment:
      DATABASE_URL: postgresql://dev:secret@db:5432/stockpro
    depends_on:
      db:
        condition: service_healthy

  frontend:
    image: 192.168.x.x:5000/stockpro-frontend:latest
    ports: ["5173:80"]
    depends_on: [backend]

  db:
    image: postgres:15-alpine
    environment: { POSTGRES_DB: stockpro, POSTGRES_USER: dev, POSTGRES_PASSWORD: secret }
    volumes: [pgdata:/var/lib/postgresql/data]
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U dev -d stockpro"]
      interval: 10s; timeout: 5s; retries: 5

volumes:
  pgdata:
terminal — รัน Production บน server อื่น
# บน server อื่น (ไม่มี source code)
docker compose -f docker-compose.prod.yml up -d
✅
ตรวจสอบ
  • ทั้ง stockpro-backend และ stockpro-frontend ปรากฏใน Registry UI
  • ลอง pull image บนเครื่องอื่น → docker compose up สำเร็จ
