🔌1. API คืออะไร?

API (Application Programming Interface) คือ "สัญญา" ระหว่าง 2 โปรแกรม — กำหนดว่าจะส่งข้อมูลรูปแบบใด และรับผลลัพธ์รูปแบบใดกลับมา

Client (Browser/App) │ │ HTTP Request: GET /api/todos │ Headers: { "Authorization": "Bearer token" } ▼ Server (Node.js + Express) ───► Database (PostgreSQL) │ │ HTTP Response: 200 OK │ Body: [{"id":1,"title":"Learn DevOps","done":false}] ▼ Client แสดงข้อมูลบนหน้าเว็บ
💡
ทำไมต้องแยก Frontend กับ Backend? การแยก concerns ทำให้พัฒนาแบบขนานได้ — Frontend dev และ Backend dev ทำงานได้พร้อมกัน โดยตกลงกันแค่เรื่อง API contract
นอกจากนี้ Backend เดียวกันยังรองรับ Web, Mobile, Desktop App ได้พร้อมกัน
📐2. REST — 6 หลักการหลัก

REST (Representational State Transfer) ไม่ใช่ protocol แต่เป็น architectural style — ชุดข้อกำหนดสำหรับออกแบบ API บน HTTP

1. Client-Server
แยก UI (client) ออกจาก data storage (server) — แต่ละฝั่งพัฒนาอิสระจากกัน
2. Stateless
แต่ละ request ต้องมีข้อมูลครบในตัว — server ไม่เก็บ session state ระหว่าง requests
3. Cacheable
Response ระบุได้ว่า cache ได้หรือไม่ — ลด latency และ server load
4. Uniform Interface
ใช้ HTTP methods (GET/POST/PUT/DELETE) ตาม convention — ทำให้ API เข้าใจง่าย
5. Layered System
Client ไม่รู้ว่าคุยกับ server จริงหรือ load balancer, proxy, CDN — แต่ละ layer ทำหน้าที่ของตัวเอง
6. Code on Demand (Optional)
Server ส่ง executable code มาให้ client รัน (เช่น JavaScript) — ข้อนี้ optional
📨3. HTTP Methods — CRUD Mapping

RESTful API ใช้ HTTP methods แทนการกระทำกับ resource — ทำให้ URL อ่านแล้วรู้ว่าทำอะไร

MethodCRUDตัวอย่าง URLคำอธิบายIdempotent?
GET Read GET /todos
GET /todos/1		 ดึงรายการ / ดึง 1 รายการ ✓ ใช่ (รันกี่ครั้งก็ได้ผลเหมือนกัน)
POST Create POST /todos สร้าง resource ใหม่ ✗ ไม่ใช่ (รันซ้ำ = สร้างซ้ำ)
PUT Replace PUT /todos/1 แทนที่ resource ทั้งหมด ✓ ใช่
PATCH Update PATCH /todos/1 แก้ไข resource บางส่วน ✓ ใช่ (ถ้าออกแบบดี)
DELETE Delete DELETE /todos/1 ลบ resource ✓ ใช่ (ลบซ้ำก็ไม่มีอยู่อยู่ดี)
💡
Idempotent คือ? การรัน operation เดิมหลายครั้งให้ผลเหมือนกับรันครั้งเดียว — เช่น DELETE /todos/1 จะลบ todo#1 ไม่ว่ารันกี่ครั้ง (ครั้งแรกลบ ครั้งต่อมาก็แค่ return 404)
📊4. HTTP Status Codes

Status code บอก client ว่าผลลัพธ์ของ request เป็นอย่างไร — แบ่งเป็น 5 กลุ่มตาม prefix

2xx
Success
200 OK · 201 Created · 204 No Content
3xx
Redirect
301 Moved Permanently · 304 Not Modified
4xx
Client Error
400 Bad Request · 401 Unauthorized · 403 Forbidden · 404 Not Found · 422 Unprocessable
5xx
Server Error
500 Internal Server Error · 502 Bad Gateway · 503 Service Unavailable
⚠️
ผิดพลาดที่พบบ่อย อย่า return 200 OK เมื่อเกิด error! — บาง API ทำแบบนี้แล้วใส่ error ไว้ใน body แทน ซึ่งทำให้ client ต้อง parse body เพิ่มเติม ควรใช้ status code ที่ถูกต้อง
⚡5. Node.js — Single-Threaded & Non-Blocking I/O

Node.js ใช้ single thread แต่รับได้หลาย request พร้อมกัน — ด้วย Event Loop ที่จัดการ async operations แบบ non-blocking

Call Stack Node.js APIs Callback Queue ┌──────────┐ ┌─────────────────┐ ┌────────────────────┐ │ main() │ ──► │ setTimeout() │ ──► │ timer callbacks │ │ readDB() │ │ fs.readFile() │ │ I/O callbacks │ │ express()│ │ fetch() │ │ HTTP callbacks │ └──────────┘ └─────────────────┘ └────────────────────┘ ▲ │ └───────────── Event Loop ◄──────────────────────┘ (ถ้า Call Stack ว่าง → ดึง callback มารัน)
Synchronous (Blocking)
รอให้ operation เสร็จก่อนทำต่อ — ถ้า query DB ใช้เวลา 2 วินาที request อื่นต้องรอ 2 วินาที
Asynchronous (Non-blocking)
ส่ง operation ไปทำ background แล้วรับ callback เมื่อเสร็จ — ระหว่างรอรับ request อื่นได้ปกติ
🧵
Node.js เหมาะกับงานอะไร? ดี: I/O intensive (API servers, real-time apps, proxies) — เพราะรอ I/O แบบ non-blocking ได้ดี
ไม่เหมาะ: CPU intensive (video encoding, image processing, ML) — เพราะ single thread จะ block ทั้งหมด
🚂6. Express.js — Middleware Pattern

Middleware คือ function ที่นั่งกลาง request กับ response — ทุก request ผ่าน middleware stack ตามลำดับ

Request ──► [logger] ──► [auth] ──► [validate] ──► [handler] ──► Response ถ้า next() ถ้า next() ถ้า next() return res.json() ไม่ถูกเรียก → หยุดที่นี่ (เช่น auth fail → 401)

🗄️7. PostgreSQL — ACID Properties

PostgreSQL เป็น Relational Database ที่รองรับ ACID — คุณสมบัติที่รับประกันความถูกต้องของข้อมูลแม้เกิดข้อผิดพลาด

A — Atomicity (ครบหรือไม่มีเลย)
Transaction ทำสำเร็จทั้งหมดหรือล้มเหลวทั้งหมด — ไม่มีสถานะกึ่งกลาง เช่น โอนเงิน: หักต้นทาง + เพิ่มปลายทาง ต้องเกิดพร้อมกัน
C — Consistency (ถูกต้องตลอด)
ข้อมูลต้องผ่าน constraints ทุกข้อก่อนและหลัง transaction — เช่น NOT NULL, UNIQUE, FOREIGN KEY
I — Isolation (แยกจากกัน)
Transactions หลายอันรันพร้อมกันไม่กวนกัน — ผลลัพธ์เหมือนรันทีละอัน
D — Durability (คงทน)
เมื่อ commit แล้ว ข้อมูลจะอยู่แม้ server ล่มหรือไฟดับ — บันทึกลง disk จริง
🆚
SQL vs NoSQL — เลือกอะไร? SQL (PostgreSQL): ข้อมูลมีโครงสร้างชัดเจน, ต้องการ ACID, queries ซับซ้อน (JOIN)
NoSQL (MongoDB): ข้อมูลยืดหยุ่น, scale horizontal ง่าย, ไม่ต้องการ schema เข้มงวด
สำหรับ Todo App ที่มี schema ชัดเจน → PostgreSQL เหมาะกว่า

Connection Pooling:

การสร้าง DB connection ใหม่ทุก request มีต้นทุนสูง (TCP handshake, auth) — Connection Pool เก็บ connections สำเร็จรูปและ reuse ได้ทันที ลด latency และ resource usage

📖8. คำศัพท์ที่ควรรู้
endpoint
URL + HTTP method คู่หนึ่ง เช่น GET /todos — แต่ละ endpoint ทำหน้าที่เฉพาะ
request body
ข้อมูลที่ client ส่งมาใน POST/PUT/PATCH — มักเป็น JSON format
response body
ข้อมูลที่ server ตอบกลับ — มักเป็น JSON ใน REST API
query parameter
ตัวกรองใน URL เช่น /todos?done=true&limit=10 — ใช้สำหรับ filter/sort/paginate
route parameter
ค่าในเส้นทาง URL เช่น /todos/:id — ดึงได้ด้วย req.params.id ใน Express
middleware
function ที่อยู่กลาง request-response cycle — ใช้สำหรับ auth, logging, validation
ORM
Object-Relational Mapper — ใช้ code แทน SQL เช่น Prisma, Sequelize (เราใช้ raw SQL ใน Week 2)
CORS
Cross-Origin Resource Sharing — browser security ที่บล็อก request ข้าม domain ต้องตั้งค่า header ฝั่ง server
