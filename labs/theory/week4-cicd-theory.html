🔁2. CI vs CD — ความแตกต่าง
🔄 CI — Continuous Integration
นักพัฒนา merge code บ่อยๆ (วันละหลายครั้ง) — ทุก merge trigger automated build และ test ให้ตรวจสอบโดยทันที ตรวจจับ integration bug ได้เร็ว
📬 CD — Continuous Delivery
Code ที่ผ่าน CI จะ พร้อม deploy อัตโนมัติเสมอ — แต่ยังต้องใช้คนกด deploy เข้า production จริง (manual approval step)
🚀 CD — Continuous Deployment
Code ที่ผ่าน CI จะ deploy อัตโนมัติ เข้า production ทันที — ไม่ต้องมีคนกด เหมาะกับ teams ที่มี test coverage สูงมาก
⚠️
"CD" มีความหมาย 2 อย่าง Continuous Delivery = พร้อมแต่คนกด · Continuous Deployment = อัตโนมัติทั้งหมด
ใน Lab นี้เราทำ Continuous Delivery — CD workflow push image ไปที่ Registry แต่ยังต้องรัน docker compose ด้วยมือบน server

⚙️4. GitHub Actions Architecture

GitHub Actions เป็น CI/CD platform ที่ built-in อยู่ใน GitHub ทุก account — ใช้ YAML ไว้ใน .github/workflows/ และ GitHub จัดการ infrastructure ให้ทั้งหมด

Workflow
ไฟล์ YAML หนึ่งไฟล์ใน .github/workflows/ — กำหนด trigger และ jobs ทั้งหมด
Event (Trigger)
สิ่งที่ทำให้ workflow รัน เช่น push, pull_request, schedule, workflow_dispatch
Job
กลุ่มของ steps ที่รันบน runner เดียวกัน — หลาย jobs รันพร้อมกันได้ (parallel)
Step
หน่วยเล็กสุดใน job — แต่ละ step รัน action หรือ shell command
Runner
VM ที่ GitHub จัดให้รัน workflow — ubuntu-latest, windows-latest, macos-latest
Action
Reusable component เช่น actions/checkout@v4 — ดาวน์โหลดจาก GitHub Marketplace
# โครงสร้าง Workflow YAML name: CI # ชื่อ workflow on: # trigger events push: branches: [develop, main] pull_request: branches: [develop] jobs: # รายการ jobs build: # ชื่อ job runs-on: ubuntu-latest # เลือก runner steps: # รายการ steps - uses: actions/checkout@v4 # ใช้ action สำเร็จรูป - name: Install deps # ชื่อ step run: npm ci # รัน shell command working-directory: ./backend
💰
GitHub Actions Free Tier Public repositories: ฟรีไม่จำกัด ✓
Private repositories: 2,000 นาที/เดือน (ubuntu) ฟรี — เพียงพอสำหรับ project ขนาดเล็ก-กลาง

🎯6. Deployment Strategies

วิธีนำ version ใหม่ขึ้น production มีหลายแบบ — แต่ละแบบมี tradeoff ระหว่าง risk, downtime, และความซับซ้อน

🔄 Rolling Deployment
v1 v1 v1 → v2 v1 v1 → v2 v2 v1 → v2 v2 v2
อัพเดต instances ทีละน้อย — ไม่มี downtime แต่ 2 versions รันพร้อมกันช่วงสั้นๆ
🔵🟢 Blue-Green Deployment
Blue(v1) [live] Green(v2) [standby] switch traffic → Green(v2) [live] Blue(v1) [standby]
มี 2 environment — switch traffic ทันที rollback ได้เร็ว แต่ต้องการ resource 2 เท่า
🐦 Canary Deployment
100% → v1 users 10% → v2 users (canary) ถ้า OK → เพิ่มเป็น 100% → v2
ค่อยๆ เพิ่ม traffic ไปหา version ใหม่ — ทดสอบกับ users จริงก่อน full rollout
💥 Recreate Deployment
v1 [running] ↓ stop v1 (downtime!) v2 [start]
หยุด version เก่าก่อนแล้วค่อยเริ่ม version ใหม่ — มี downtime แต่ simple ที่สุด เหมาะกับ dev/staging
👁️7. Code Review — Best Practices

Code Review ที่ดีไม่ใช่แค่ตรวจหา bugs — มันเป็นกระบวนการ knowledge sharing และรักษา code quality ของทีม

Author (ผู้เปิด PR)
✓ PR ขนาดเล็ก (ไม่เกิน 400 บรรทัด)
✓ เขียน description ชัดเจน
✓ ทดสอบก่อน request review
✓ ตอบ comment ทุกข้อ
Reviewer (ผู้ review)
✓ Review ภายใน 24 ชั่วโมง
✓ Comment เชิงสร้างสรรค์ มีเหตุผล
✓ แยก blocking vs non-blocking issues
✓ Approve เมื่อพร้อม — ไม่ delay โดยไม่มีเหตุ
💬
Comment ที่ดี vs ไม่ดี ❌ "ทำไม่ถูก" — ไม่มีเหตุผล สร้าง friction
✓ "บรรทัดนี้อาจ throw error ถ้า users เป็น null เพราะ... แนะนำให้ add null check ก่อน access users.id"

ใช้ prefix: nit: (minor) · blocking: (ต้องแก้ก่อน merge) · question: (แค่สงสัย)
📖8. คำศัพท์ที่ควรรู้
pipeline
ลำดับขั้นตอนอัตโนมัติที่ code ต้องผ่านตั้งแต่ push จนถึง deploy
runner
VM หรือ server ที่รัน GitHub Actions jobs — GitHub จัดให้หรือเป็น self-hosted
workflow trigger
event ที่ทำให้ workflow รัน เช่น push, pull_request, schedule (cron)
artifact
ผลลัพธ์ของ build step เช่น Docker image, binary file, test report
secret
ค่าลับที่ encrypt เก็บใน GitHub Settings — ใช้ใน workflow ผ่าน secrets.NAME
environment
กลุ่มของ server ที่ deploy ลงไป เช่น development, staging, production
branch protection
กฎที่บังคับ workflow ก่อน merge — เช่น ต้อง review, ต้องผ่าน CI
semantic versioning
รูปแบบ version เช่น v1.2.3 = MAJOR.MINOR.PATCH — MAJOR: breaking, MINOR: feature, PATCH: fix
CHANGELOG
ไฟล์บันทึกการเปลี่ยนแปลงในแต่ละ release — generate อัตโนมัติจาก conventional commits ได้
rollback
การย้อนกลับไปใช้ version ก่อนหน้า เมื่อ deployment ใหม่มีปัญหา
