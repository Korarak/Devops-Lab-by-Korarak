⚖️1. VM vs Container — ความแตกต่างหลัก

ทั้งสองเป็น isolation technology แต่ระดับที่ต่างกัน — VM จำลอง hardware ทั้งหมด ส่วน Container แชร์ OS kernel ของ host

🖥️ Virtual Machine (VM)
  • มี OS เต็มๆ ของตัวเอง (Guest OS)
  • ขนาดใหญ่ — หลาย GB ต่อ VM
  • Boot ช้า — หลายนาที
  • Strong isolation — แยกขาดจากกัน
  • ใช้ Hypervisor (VMware, VirtualBox, KVM)
📦 Container (Docker)
  • แชร์ Linux kernel ของ host
  • ขนาดเล็ก — ไม่กี่ร้อย MB
  • Start ได้ใน milliseconds
  • Process-level isolation ด้วย namespaces + cgroups
  • ใช้ Docker Engine หรือ containerd
🤝
ใช้ร่วมกันได้! ในชีวิตจริง container มักรันอยู่ใน VM — เช่น Proxmox VM ที่รัน Docker containers อยู่ด้านใน ได้ประโยชน์ทั้ง isolation ของ VM และความเร็วของ containers
🏗️2. Docker Architecture — Client-Server

Docker ทำงานแบบ client-server — เราพิมพ์คำสั่งผ่าน Docker CLI (client) ซึ่งส่งไปให้ Docker Daemon (server) ทำงาน

Docker CLI Docker Daemon (dockerd) (docker build, run, push) ┌───────────────────────────────┐ │ │ │ │ REST API over Unix socket │ ┌─────────┐ ┌──────────┐ │ └────────────────────────────► │ │ Images │ │Containers│ │ │ └─────────┘ └──────────┘ │ │ ┌─────────┐ ┌──────────┐ │ │ │Volumes │ │ Networks │ │ │ └─────────┘ └──────────┘ │ └───────────┬───────────────────┘ │ pull/push ▼ Registry (Docker Hub / Private)
Docker Daemon (dockerd)
Service ที่รันอยู่เบื้องหลัง จัดการ images, containers, networks, volumes — ฟัง Unix socket หรือ TCP
Docker CLI
Command-line tool ที่เราใช้พิมพ์คำสั่ง — แค่ client ส่ง command ไปให้ daemon ทำงาน
Registry
ที่เก็บ Docker images — Docker Hub เป็น public registry ส่วน Private Registry อยู่บน server ของเรา
📸3. Image vs Container — Class กับ Instance
🖼️ Docker Image (แม่พิมพ์)
Read-only template ที่มี OS, dependencies, app code — เหมือน "class" ใน OOP ไม่สามารถแก้ไขได้หลัง build
📦 Docker Container (ของจริง)
Instance ของ Image ที่กำลังรันอยู่ — เหมือน "object" ที่ new มาจาก class มี state ของตัวเอง สร้างได้หลาย containers จาก image เดียว
🎯
ตัวอย่างง่ายๆ postgres:15 คือ Image — สามารถ run เป็น Container db1, db2, db3 ได้พร้อมกัน โดยแต่ละอันมีข้อมูลต่างกัน
เหมือน recipe ที่ทำอาหารได้หลายจาน recipe เดียวกัน แต่ละจานต่างกัน
🥞4. Dockerfile & Image Layers

Docker Image ประกอบด้วย layers — แต่ละ instruction ใน Dockerfile สร้าง layer ใหม่ layers ที่ไม่เปลี่ยนจะถูก cache ไว้ทำให้ build ครั้งต่อไปเร็วขึ้น

Layer 5: COPY app files (เปลี่ยนบ่อย)
Layer 4: npm ci (เปลี่ยนเมื่อ package.json เปลี่ยน)
Layer 3: COPY package*.json
Layer 2: RUN apt-get install curl
Layer 1: FROM node:20-alpine
Shared: Linux Kernel (Host)
↑ layers ล่างเปลี่ยนน้อย → cache นานกว่า
⚡
Layer Caching Strategy วาง instructions ที่ "เปลี่ยนน้อย" ไว้ด้านล่าง และ "เปลี่ยนบ่อย" ไว้ด้านบน

✓ COPY package.json ก่อน → npm ci → COPY source code
ถ้าแค่แก้ source code → Docker cache layer ของ npm ci ไว้ ไม่ต้อง install ซ้ำ

Multi-stage Build — ลดขนาด Image:

# Stage 1: Build (ใช้ image เต็ม) FROM node:20 AS builder WORKDIR /app COPY . . RUN npm ci && npm run build # Stage 2: Production (ใช้ image เล็ก — ไม่มี dev tools) FROM nginx:alpine COPY --from=builder /app/dist /usr/share/nginx/html # ผลลัพธ์: image เล็กลงจาก ~1.2GB → ~25MB!
🌐5. Docker Networking

Docker มี network drivers ต่างๆ สำหรับ use cases ต่างกัน — การเชื่อมต่อระหว่าง containers ใช้ชื่อ service แทน IP address

Network Typeลักษณะใช้เมื่อ
bridge (default) Virtual network สำหรับ containers บน host เดียวกัน — containers คุยกันได้ผ่านชื่อ ใช้บ่อยที่สุด เช่น frontend คุยกับ backend
host Container ใช้ network stack ของ host โดยตรง — ไม่มี isolation แต่เร็วกว่า ต้องการ performance สูงสุด
none Container ไม่มี network เลย Security สูงสุด ไม่ต้องการ network
overlay เชื่อม containers ข้าม hosts หลายเครื่อง Docker Swarm / Kubernetes
💡
Container DNS ใน Docker Compose ใน docker-compose.yml ถ้า service ชื่อ backend — container อื่นเรียกได้ด้วย URL http://backend:3000 โดยไม่ต้องรู้ IP address
Docker จัดการ DNS resolution อัตโนมัติภายใน network เดียวกัน
💾6. Volumes — ข้อมูลถาวรใน Container

Container เป็น ephemeral — ถ้า container ถูกลบ ข้อมูลภายในหายทั้งหมด Volume คือทางออกในการ persist ข้อมูล

🗂️ Named Volume
Docker จัดการเอง เก็บใน /var/lib/docker/volumes/ — เหมาะสำหรับ database data, persistent state
📁 Bind Mount
Map folder ของ host เข้า container โดยตรง — เหมาะสำหรับ development (hot reload) เพราะแก้ไขได้ทันที
🧠 tmpfs Mount
เก็บข้อมูลใน RAM เท่านั้น — ข้อมูลหายเมื่อ container หยุด เหมาะสำหรับ sensitive data ชั่วคราว
# Named Volume — ข้อมูล DB ไม่หายแม้ลบ container volumes: postgres_data: # Docker สร้าง volume ชื่อนี้ services: db: image: postgres:15 volumes: - postgres_data:/var/lib/postgresql/data # mount volume # Bind Mount — sync code แบบ real-time ตอน dev backend: volumes: - ./backend:/app # folder host : folder ใน container
🎼7. Docker Compose — Multi-Container Orchestration

Docker Compose จัดการหลาย containers พร้อมกัน — กำหนด services, networks, volumes ใน YAML ไฟล์เดียว แล้วรันด้วยคำสั่งเดียว

# docker-compose.yml structure services: frontend: # ชื่อ service = ชื่อ DNS ใน network build: ./frontend ports: ["5173:80"] # host:container depends_on: [backend] backend: build: ./backend ports: ["3000:3000"] environment: - DATABASE_URL=postgresql://user:pass@db:5432/mydb depends_on: [db] db: image: postgres:15 volumes: - pgdata:/var/lib/postgresql/data environment: POSTGRES_PASSWORD: secret volumes: pgdata:
คำสั่งทำอะไร
docker compose up -dBuild + Start ทุก service ใน background
docker compose downStop + Remove containers (volume ยังอยู่)
docker compose down -vStop + Remove containers + volumes
docker compose logs -fดู logs realtime ของทุก service
docker compose psดูสถานะทุก service
docker compose exec backend shเปิด shell เข้าไปใน container
📖8. คำศัพท์ที่ควรรู้
image
Read-only template สำหรับสร้าง container — build จาก Dockerfile หรือ pull จาก Registry
container
Instance ที่รันจาก image — มี filesystem, network, process isolation ของตัวเอง
Dockerfile
ไฟล์ instructions สำหรับ build image — แต่ละ instruction สร้าง layer
registry
ที่เก็บและแจกจ่าย images เช่น Docker Hub, Private Registry
tag
version label ของ image เช่น node:20, postgres:15, myapp:v1.0.0
port mapping
เชื่อม port ของ host กับ port ของ container เช่น -p 8080:80 (host:container)
environment variable
ตัวแปร config ที่ส่งเข้า container เช่น DATABASE_URL — ใช้แทนการ hardcode
health check
คำสั่งที่ Docker รันซ้ำๆ เพื่อตรวจว่า container ทำงานปกติ — บอก orchestrator ว่า restart หรือไม่
namespace
Linux kernel feature ที่ isolate containers จากกัน — PID, Network, Mount, UTS namespaces
cgroups
Linux kernel feature ที่จำกัด CPU, Memory ที่ container ใช้ได้
