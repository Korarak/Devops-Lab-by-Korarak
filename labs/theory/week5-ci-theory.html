🔄 1. หลักการ Continuous Integration (CI)

Continuous Integration ถูกเสนอโดย Martin Fowler และ Kent Beck ในปี 1990s เป็น engineering practice ที่ให้ developer integrate code เข้า shared branch บ่อย ๆ (อย่างน้อยวันละครั้ง) และตรวจสอบอัตโนมัติทุกครั้ง

📖
คำนิยาม (Martin Fowler) "Continuous Integration is a software development practice where members of a team integrate their work frequently... Each integration is verified by an automated build (including test) to detect integration errors as quickly as possible."

8 หลักการของ CI (Fowler's CI Practices)

1
Maintain a Single Source Repository
code ทั้งทีมอยู่บน shared repository เดียว (Git) มีแค่ branch เดียวเป็น "source of truth" (main/trunk)
2
Automate the Build
build ต้องรันได้ด้วย single command — ไม่มี manual steps ซ่อนอยู่
3
Make Your Build Self-Testing
build ต้องรัน automated tests — ถ้า test fail = build fail
4
Everyone Commits to Mainline Every Day
push บ่อย ๆ → integration conflict น้อยลง → merge ง่ายขึ้น
5
Every Commit Should Build Mainline
ทุก commit ที่เข้า main branch จะถูก build อัตโนมัติทันที
6
Keep the Build Fast
build ควรเสร็จภายใน 10 นาที — ถ้านานกว่านั้น developer จะหลีกเลี่ยงการ push บ่อย ๆ
7
Test in a Clone of the Production Environment
CI environment ควรใกล้เคียง production มากที่สุด — ใช้ Docker ช่วยได้
8
Make It Easy to Get the Latest Deliverable
ทุกคนควร build ล่าสุดได้เสมอ — ผ่าน artifact หรือ package registry
⚙️ 2. GitHub Actions Architecture

GitHub Actions เป็น CI/CD platform ที่ built-in อยู่ใน GitHub ทำงานโดยรับ Events จาก repository แล้วรัน Workflows ที่กำหนดไว้

Component ทั้งหมด

Event (push, PR, schedule, ...)
  ↓
Workflow (.github/workflows/*.yml)
  ↓
Job (กลุ่มงาน — รันบน Runner แยกกัน)
  ├── Step 1: uses: actions/checkout@v4
  ├── Step 2: uses: actions/setup-node@v4
  └── Step 3: run: npm test

Runner (VM ที่ GitHub จัดให้ หรือ self-hosted)
  └── ubuntu-latest, windows-latest, macos-latest

Action (reusable code unit — published บน GitHub Marketplace)
  └── actions/checkout, actions/setup-node, docker/build-push-action

Contexts และ Expressions

Contextตัวแปรที่ใช้บ่อยตัวอย่างค่า
githubgithub.sha, github.ref_name, github.actor, github.event_namea3f2c1d..., main, username, push
envVariables ที่ตั้งใน env: block${{ env.NODE_ENV }} → test
secretsSecrets จาก repo/org settings${{ secrets.REGISTRY_PASS }} → ****
matrixค่าจาก strategy.matrix${{ matrix.node-version }} → 20
stepsOutput จาก step ก่อนหน้า${{ steps.myid.outputs.result }}

Job Dependencies

# Parallel (default) — ทั้ง lint และ test รันพร้อมกัน
jobs:
  lint: ...
  test: ... ← รันพร้อม lint ไม่รอ

# Sequential — test รอให้ lint ผ่านก่อน
jobs:
  lint: ...
  test:
    needs: lint ← เพิ่ม needs: เพื่อสร้าง dependency

# Fan-out — หลาย jobs รอ job เดียว
jobs:
  build: ...
  deploy-staging: { needs: build }
  deploy-prod: { needs: build }

🧪 3. Testing Pyramid

Testing Pyramid เสนอโดย Mike Cohn (2009) อธิบายว่า test ควรมีสัดส่วนอย่างไร — ฐานกว้าง (unit test เยอะ) ยอดแหลม (E2E test น้อย)

น้อย / ช้า / แพง
E2E Tests
Cypress, Playwright
ทดสอบ user flow ทั้งหมด บน real browser
Integration Tests
Supertest + real DB
ทดสอบ component หลายตัวทำงานร่วมกัน
Unit Tests
Jest + mock
ทดสอบ function/module เดียว ๆ แยกกัน

โครงสร้าง Unit Test ที่ดี (AAA Pattern)

describe('GET /api/stocks', () => {
  test('returns 200 with array', async () => {

    // ARRANGE — เตรียม data ที่ต้องการ
    db.query.mockResolvedValueOnce({ rows: [
      { id: 1, name: 'Rice', quantity: 100 }
    ]});

    // ACT — เรียก function/endpoint ที่ต้องการทดสอบ
    const res = await request(app).get('/api/stocks');

    // ASSERT — ตรวจสอบว่าผลลัพธ์ถูกต้อง
    expect(res.statusCode).toBe(200);
    expect(Array.isArray(res.body)).toBe(true);
  });
});

Jest Matchers ที่ใช้บ่อย

Matcherใช้เมื่อตัวอย่าง
toBe(value)เปรียบเทียบแบบ strict (===)expect(2+2).toBe(4)
toEqual(value)เปรียบเทียบ object/array ทุก propertyexpect(obj).toEqual({ a: 1 })
toHaveProperty(key, val?)ตรวจ property ใน objectexpect(res.body).toHaveProperty('id')
toHaveLength(n)ตรวจความยาว array/stringexpect(arr).toHaveLength(3)
toBeGreaterThan(n)ตรวจว่า > nexpect(uptime).toBeGreaterThan(0)
toBeNull() / toBeTruthy()ตรวจ null/truthy valueexpect(user).not.toBeNull()
toThrow()ตรวจว่า function throw errorexpect(() => fn()).toThrow('Error')

Code Coverage Metrics

Statement Coverage
% ของ statement ที่รันผ่านระหว่าง test — metric ทั่วไปที่สุด
Branch Coverage
% ของ if/else/switch ที่ครอบคลุมทั้ง true และ false — เข้มกว่า statement
Function Coverage
% ของ function ที่ถูกเรียก — ถ้าน้อย แสดงว่ามี dead code
Line Coverage
% ของบรรทัดที่ execute — คล้าย statement แต่นับทีละ line
⚠️
100% Coverage ≠ Bug-free Coverage วัดแค่ว่า code "ถูกรัน" ไม่ใช่ว่าถูก test อย่างครบถ้วน test ที่ดีต้องมี meaningful assertions ไม่ใช่แค่ให้ coverage ผ่าน
🔍 4. Static Analysis กับ ESLint

Static Analysis คือการวิเคราะห์ code โดยไม่ต้องรัน — หา bug, style inconsistency, และ potential issues ก่อนที่ code จะ execute

Linting (ESLint)
ตรวจ syntax, style, และ potential bugs — ตัวอย่าง: no-unused-vars, eqeqeq, semi
Formatting (Prettier)
จัด code ให้สวยงามสม่ำเสมอ — indent, quotes, line length (ทำงานร่วมกับ ESLint ได้)
SAST (Semgrep)
Security static analysis — ตรวจหา XSS, SQL injection, hardcoded secrets (เนื้อหา Week 15)
Type Check (TypeScript)
ตรวจ type ณ compile time — จับ type mismatch ก่อนรัน (ใช้แทน ESLint บางส่วนได้)

ESLint Rules ที่ควรรู้

Ruleความหมายระดับ
no-unused-varsประกาศ variable แต่ไม่ใช้warn
eqeqeqต้องใช้ === แทน ==error
no-varห้ามใช้ var (ใช้ let/const แทน)error
prefer-constใช้ const เมื่อไม่ได้ reassignwarn
semiต้องมี ; ท้ายทุก statementerror
no-consoleห้ามใช้ console.log ใน production codewarn/off
✅ 5. CI Best Practices
🚀 Fast Feedback
CI ควรเสร็จภายใน 10 นาที ใช้ cache (node_modules), parallel jobs, และ skip unnecessary jobs
🔒 Fail Fast
Lint ก่อน Test ก่อน Build — ถ้า lint fail ไม่ต้องรัน test เสียเวลา ใช้ needs:
🌿 Branch Strategy
รัน CI บน feature/* ทุกครั้ง · Branch Protection บน main/develop ต้องการ CI pass
🗄️ Artifact Retention
เก็บ coverage report, build log ไว้ 7-14 วัน เพื่อ debug ภายหลัง ไม่ต้องเก็บนานเกิน
🔑 Secrets Management
ห้าม hardcode credentials ใน workflow ใช้ secrets.* เสมอ ตรวจ audit log เป็นประจำ
🚫 No Flaky Tests
Test ที่ผ่านบางครั้ง fail บางครั้ง (flaky) ทำลายความเชื่อถือ CI ต้องแก้ก่อนเสมอ
💡
npm ci vs npm install ใน CI ให้ใช้ npm ci เสมอ — มันอ่านจาก package-lock.json ทำให้ environment reproducible และเร็วกว่า npm install ด้วย
