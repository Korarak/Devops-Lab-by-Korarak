Code Quality Metrics, SAST & Image Tagging

SonarQube · Quality Gate · Static Analysis · Docker Image Tagging Strategies

📊 Code Quality คืออะไร

Code Quality คือการวัดว่า source code มีคุณภาพดีแค่ไหนในมิติต่างๆ ทั้ง correctness, maintainability, security และ reliability

มิติของ Code Quality

🐛
Reliability
ไม่มี Bug ที่ทำให้ app crash หรือ behave ผิดพลาด
🔒
Security
ไม่มี Vulnerability ที่ผู้โจมตีสามารถใช้ประโยชน์ได้
🔧
Maintainability
Code อ่านง่าย ไม่ซับซ้อนเกินไป ไม่มี Code Smells
📊
Coverage
% ของ code ที่ถูกทดสอบด้วย unit/integration tests
🔁
Duplications
ปริมาณ code ที่ซ้ำกัน (ยิ่งน้อยยิ่งดี)

🔬 SAST — Static Application Security Testing

SAST (Static Analysis) วิเคราะห์ source code โดยไม่ต้องรัน — ค้นหา bug, vulnerability และ code smell จากการอ่าน code อย่างเดียว

🚦 Quality Gate — กลไกการควบคุมคุณภาพ

Quality Gate คือชุดของ Conditions (เงื่อนไข) ที่กำหนดว่า code ต้องผ่านอะไรบ้างก่อนถือว่า "พร้อม release"

ตัวอย่าง Quality Gate Conditions

MetricConditionเหตุผล
Coverage≥ 60%ต้องมี test ครอบคลุมพอสมควร
Duplicated Lines< 30%หลีกเลี่ยง copy-paste code
Reliability Rating≥ Aไม่มี Bug ระดับ Critical/Major
Security Rating≥ Aไม่มี Vulnerability
Security Hotspots Reviewed= 100%Hotspots ทั้งหมดถูก review

New Code vs Overall Code

💡
Default Sonar Way Gate ตรวจเฉพาะ "New Code" SonarQube นิยาม "New Code" ว่าคือ code ที่เพิ่มหรือแก้ตั้งแต่วันที่กำหนด (เช่น 30 วันล่าสุด) — ทำให้ team ที่มี legacy code ไม่ถูก block ด้วยปัญหาเก่า แต่ต้องรักษาคุณภาพ code ใหม่

🏷️ Docker Image Tagging Strategies

Tag คือ label ที่ใช้ระบุ version ของ Docker Image — Strategy ที่เลือกมีผลต่อ traceability, rollback และ deployment safety

Strategy ต่างๆ

Strategyตัวอย่าง Tagข้อดีข้อเสีย
latest only:latestเรียบง่ายไม่รู้ว่า image ไหนถูก deploy
Commit SHA:sha-a3f7c91trace ได้แม่นยำต้องรู้ SHA ก่อน pull
Semantic Version:v1.2.3human-readable, stableต้อง tag manually
Branch + SHA:develop-abc1234รู้ branch + committag ยาว
SHA + latest (Week 6):sha-a3f7c91 + :latestสมดุลระหว่าง trace กับ ง่ายใช้ต้องจัดการสองตัว

git rev-parse --short HEAD

# ดู full SHA $ git rev-parse HEAD a3f7c91b2d4e8f0c1234567890abcdef12345678 # ดู short SHA (7 ตัว — unique พอสำหรับงานทั่วไป) $ git rev-parse --short HEAD a3f7c91 # ใน GitHub Actions run: echo "sha_short=$(git rev-parse --short HEAD)" >> $GITHUB_OUTPUT
⚠️
ระวัง: :latest ไม่ใช่ "ล่าสุดเสมอ" :latest tag ถูก overwrite ทุกครั้งที่ push — ถ้า push หลาย version พร้อมกัน อาจมี race condition ที่ :latest ชี้ไปที่ version เก่ากว่า ใน production ควรใช้ specific tag (SHA หรือ SemVer) เสมอ

⚡ Docker BuildKit & Layer Cache

BuildKit คือ build engine รุ่นใหม่ของ Docker ที่มี parallel build, advanced caching และ secrets handling ที่ปลอดภัยกว่า

Layer Cache ทำงานอย่างไร

# Dockerfile layer analysis FROM node:20-alpine ← Layer 1: cache hit เกือบทุกครั้ง (image ไม่เปลี่ยน) WORKDIR /app ← Layer 2: cache hit (ไม่เปลี่ยน) COPY package*.json ./ ← Layer 3: cache MISS ถ้า package.json เปลี่ยน RUN npm ci --omit=dev ← Layer 4: cache MISS ถ้า Layer 3 miss (npm install ใหม่) COPY . . ← Layer 5: cache MISS เกือบทุกครั้ง (code เปลี่ยน) CMD ["node", "index.js"] ← Layer 6: cache hit ถ้า Layer 5 ไม่เปลี่ยน CMD
✅
Best Practice: แยก COPY package.json ออกมาก่อน การ COPY package*.json ./ ก่อน RUN npm ci แล้วค่อย COPY . . ทีหลัง ทำให้ Docker cache layer ของ npm install ได้ — ถ้าแค่ code เปลี่ยน (ไม่เปลี่ยน package.json) จะไม่ต้อง npm install ใหม่

GitHub Actions Cache (type=gha)

Parameterความหมาย
cache-from: type=gharestore Docker layer cache จาก GitHub Actions cache store
cache-to: type=gha,mode=maxsave ทุก layer กลับเข้า cache (mode=max = เก็บทุก intermediate layer)
