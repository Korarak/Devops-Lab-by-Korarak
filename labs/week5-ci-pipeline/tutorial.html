GitHub Actions Architecture — ลึกกว่าเดิม
⏱ 20 นาที

ก่อนเขียน workflow ขอทบทวนและขยายความเข้าใจ GitHub Actions ให้ลึกขึ้น เพื่อให้ออกแบบ pipeline ได้อย่างถูกต้อง

โครงสร้าง Workflow YAML

name: CI Pipeline # ชื่อ workflow (แสดงใน GitHub UI)

on: # 1. EVENTS — อะไรทำให้ workflow ทำงาน
  push:
    branches: [main, develop, 'feature/**']
  pull_request:
    branches: [main, develop]

jobs: # 2. JOBS — งานที่ต้องทำ (รันบน Runner แต่ละตัว)
  lint: # job id (ตั้งชื่ออะไรก็ได้)
    runs-on: ubuntu-latest # Runner type
    steps: # 3. STEPS — คำสั่งย่อย ๆ ในแต่ละ job
      - uses: actions/checkout@v4 # Action สำเร็จรูป
      - name: Run lint
        run: npm run lint # Shell command

  test:
    needs: lint # รอให้ lint ผ่านก่อน
    runs-on: ubuntu-latest
    steps: ...

Contexts และ Expressions ที่ใช้บ่อย

Expressionความหมายตัวอย่าง
${{ github.sha }}SHA ของ commit ปัจจุบัน (40 ตัวอักษร)ใช้ tag Docker image
${{ github.ref_name }}ชื่อ branch หรือ tagfeature/login, main, v1.0.0
${{ github.event_name }}ประเภท event ที่ triggerpush, pull_request
${{ secrets.MY_SECRET }}ค่า Secret ที่ตั้งใน repoRegistry password, SSH key
${{ env.MY_VAR }}Environment variable ที่ตั้งใน workflowAPP_VERSION, NODE_ENV
${{ matrix.node-version }}ค่าจาก matrix strategy18, 20 (เมื่อใช้ matrix build)

Runner Types

ubuntu-latest
ใช้บ่อยที่สุด เร็ว ฟรี 2,000 นาที/เดือน (public repo ฟรีไม่จำกัด)
windows-latest
สำหรับ .NET, PowerShell — ใช้ credit เร็วกว่า linux ×2
macos-latest
สำหรับ iOS/macOS build — ใช้ credit เร็วที่สุด ×10
Self-hosted Runner
ใช้เครื่องหรือ VM ของตัวเอง — ฟรี เหมาะกับ Proxmox ในห้อง lab
💡
Tip: Jobs รันแบบ Parallel โดย default ถ้าไม่ใส่ needs: jobs จะรันพร้อมกันทั้งหมด ซึ่งเร็วกว่า แต่ถ้าต้องการให้ test รันหลัง lint ผ่านเท่านั้น ต้องใส่ needs: lint
✅ เช็คความเข้าใจ
  • บอกได้ว่า Event, Job, Step ต่างกันอย่างไร
  • รู้ว่า needs: ทำให้ jobs ทำงานแบบ sequential
  • เข้าใจว่า ${{ github.sha }} ดึงค่าจาก context

Hello CI — ทดลองสร้าง Workflow แรก
⏱ 20 นาที

ก่อนสร้าง CI จริง ลองสร้าง workflow ง่าย ๆ เพื่อทำความเข้าใจ cycle การทำงาน: เขียน YAML → push → ดูผล → แก้ไข

สร้างไฟล์ hello.yml

terminal (ใน root โฟลเดอร์ stockpro)
git checkout develop
git checkout -b feature/add-ci-tests

mkdir -p .github/workflows
.github/workflows/hello.yml
name: Hello CI

on:
  push:
    branches: ['**']   # trigger ทุก branch

jobs:
  greet:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Say Hello
        run: |
          echo "Hello CI from $RUNNER_OS!"
          echo "Triggered by: ${{ github.event_name }}"
          echo "Branch: ${{ github.ref_name }}"
          echo "Commit: ${{ github.sha }}"
          echo "Actor: ${{ github.actor }}"

      - name: List files
        run: ls -la

      - name: Show Node version
        run: node --version || echo "Node not installed on runner"

Push แล้วดูผล

terminal
git add .github/workflows/hello.yml
git commit -m "ci: add hello workflow for learning"
git push -u origin feature/add-ci-tests

ไปที่ GitHub → tab Actions → เลือก workflow "Hello CI" → คลิก run ล่าสุด → คลิก job "greet" → ดู output ของแต่ละ step

✅
สิ่งที่ควรเห็นใน output Hello CI from Linux! · ชื่อ branch · SHA ยาว 40 ตัว · ชื่อ GitHub ของคุณ
⚠️
hello.yml นี้เป็นแค่ตัวทดสอบ ไฟล์นี้จะลบออกทีหลัง ไม่ต้อง merge เข้า develop
✅ Checkpoint
  • ใน Actions tab เห็น workflow "Hello CI" สีเขียว ✓
  • ใน step "Say Hello" เห็น "Hello CI from Linux!" ใน log
  • เข้าใจวิธีอ่าน log แต่ละ step ใน GitHub Actions UI

Trigger Events ที่ใช้บ่อยใน CI
⏱ 15 นาที

การเลือก trigger event ที่ถูกต้องสำคัญมาก — trigger ถี่เกินไปเปลือง minutes, triggerน้อยเกินไปก็ไม่รู้เมื่อ code พัง

Eventเมื่อไหร่ triggerใช้สำหรับ
pushทุกครั้งที่ push ขึ้น branchCI รัน lint+test เมื่อ dev push code
pull_requestเปิด PR, push ใหม่เข้า PR, syncตรวจสอบก่อน merge
workflow_dispatchกดปุ่ม Run workflow ใน UIรัน manual เมื่อต้องการ
scheduleเวลา cron ที่กำหนดNightly build, security scan
releaseสร้าง GitHub ReleaseCD — deploy เมื่อ release

ตัวอย่าง: Filter branch ด้วย paths

ตัวอย่าง trigger config ที่ใช้บ่อย
on:
  push:
    branches:
      - main
      - develop
      - 'feature/**'
      - 'hotfix/**'
    paths:           # trigger เฉพาะเมื่อไฟล์เหล่านี้เปลี่ยน
      - 'backend/**'
      - '.github/workflows/**'
  pull_request:
    branches:
      - main
      - develop
    types:           # trigger เฉพาะ event types เหล่านี้
      - opened
      - synchronize  # push ใหม่เข้า PR
      - reopened
  workflow_dispatch:  # เพิ่มปุ่ม Run workflow ใน UI
    inputs:
      environment:
        description: 'Target environment'
        required: true
        default: 'staging'
        type: choice
        options: [staging, production]
💡
ใช้ paths: เพื่อประหยัด minutes ถ้า push แค่แก้ frontend CSS ไม่ควรรัน backend test ใช้ paths: ['backend/**'] เพื่อ trigger เฉพาะเมื่อ backend เปลี่ยน

แก้ backend/index.js — เพิ่ม module.exports

เพื่อให้ Supertest สามารถ import app มาทดสอบได้โดยไม่ต้องเชื่อม port จริง ต้องแยก app ออกจาก server

backend/index.js — บรรทัดสุดท้าย (เปลี่ยนจาก app.listen)
// เดิม: app.listen(PORT, ...) อย่างเดียว
// เปลี่ยนเป็น:

const PORT = process.env.PORT || 3000;

// รัน server เฉพาะเมื่อไม่ได้ถูก require จาก test
if (require.main === module) {
  app.listen(PORT, () => {
    console.log(`StockPro backend running on port ${PORT}`);
  });
}

module.exports = app;  // ← เพิ่มบรรทัดนี้
ℹ️
ทำไมต้อง require.main === module? เมื่อ Jest require('./index.js') เพื่อทดสอบ เราไม่ต้องการให้ server เริ่มรันจริง ๆ ด้วย port — require.main === module คืนค่า false เมื่อถูก require จาก file อื่น

Unit Test 1 — ทดสอบ /health Endpoint
⏱ 20 นาที

เริ่มจาก test ที่ง่ายที่สุด — ทดสอบว่า GET /health ตอบกลับ 200 OK และมี field ที่ถูกต้อง

backend/tests/health.test.js
const request = require('supertest');
const app = require('../index');

describe('GET /health', () => {
  test('returns 200 with status ok', async () => {
    const res = await request(app).get('/health');
    expect(res.statusCode).toBe(200);
    expect(res.body).toHaveProperty('status', 'ok');
  });

  test('returns version field', async () => {
    const res = await request(app).get('/health');
    expect(res.body).toHaveProperty('version');
    expect(typeof res.body.version).toBe('string');
  });

  test('returns uptime as a number', async () => {
    const res = await request(app).get('/health');
    expect(res.body).toHaveProperty('uptime');
    expect(typeof res.body.uptime).toBe('number');
    expect(res.body.uptime).toBeGreaterThan(0);
  });
});

Unit Test 2 — ทดสอบ Products CRUD Endpoints
⏱ 25 นาที

ทดสอบ endpoint หลักของ StockPro โดยใช้ Mock สำหรับ database เพื่อให้ test รันได้โดยไม่ต้องเชื่อม PostgreSQL จริง

backend/tests/products.test.js
const request = require('supertest');

// Mock database ก่อน require app
jest.mock('../db', () => ({
  query: jest.fn(),
}));
const db = require('../db');
const app = require('../index');

describe('GET /api/products', () => {
  beforeEach(() => {
    jest.clearAllMocks();
  });

  test('returns 200 with array of products', async () => {
    db.query.mockResolvedValueOnce({
      rows: [
        { id: 1, name: 'MacBook Air M3', category: 'Electronics', price: '44900.00', stock: 15 },
        { id: 2, name: 'iPhone 16 Pro',  category: 'Electronics', price: '42900.00', stock: 8  },
      ],
    });

    const res = await request(app).get('/api/products');
    expect(res.statusCode).toBe(200);
    expect(Array.isArray(res.body)).toBe(true);
    expect(res.body).toHaveLength(2);
    expect(res.body[0]).toHaveProperty('name', 'MacBook Air M3');
  });

  test('returns empty array when no products', async () => {
    db.query.mockResolvedValueOnce({ rows: [] });
    const res = await request(app).get('/api/products');
    expect(res.statusCode).toBe(200);
    expect(res.body).toEqual([]);
  });
});

describe('POST /api/products', () => {
  beforeEach(() => {
    jest.clearAllMocks();
  });

  test('creates product and returns 201 with id', async () => {
    db.query.mockResolvedValueOnce({
      rows: [{ id: 3, name: 'Nike Air Max', category: 'Footwear', price: '4590.00', stock: 32 }],
    });

    const res = await request(app)
      .post('/api/products')
      .send({ name: 'Nike Air Max', category: 'Footwear', price: 4590, stock: 32 });

    expect(res.statusCode).toBe(201);
    expect(res.body).toHaveProperty('id');
    expect(res.body.name).toBe('Nike Air Max');
  });
});
ℹ️
jest.mock() คืออะไร? แทนที่ module จริง (db.js ที่เชื่อม PostgreSQL) ด้วย mock object ทำให้ test รันบน CI ได้โดยไม่ต้องมี database จริง — mockResolvedValueOnce กำหนดว่า mock จะ return อะไรในการเรียกครั้งนั้น ๆ

Code Coverage วัดว่า test ของเรา "วิ่งผ่าน" code กี่ % ถ้า coverage ต่ำแสดงว่ามี code path ที่ไม่ได้ทดสอบ อาจซ่อน bug ไว้

Setup ESLint — Code Quality Gate
⏱ 20 นาที

ESLint วิเคราะห์ code แบบ static — หาข้อผิดพลาดและ style ที่ไม่สม่ำเสมอ โดยไม่ต้องรัน code เป็น "gate แรก" ของ CI

สร้าง .eslintrc.json

backend/.eslintrc.json
{
  "env": {
    "node": true,
    "es2021": true,
    "jest": true
  },
  "extends": "eslint:recommended",
  "parserOptions": {
    "ecmaVersion": 2021
  },
  "rules": {
    "no-unused-vars": "warn",
    "no-console": "off",
    "semi": ["error", "always"],
    "quotes": ["warn", "single"],
    "eqeqeq": ["error", "always"],
    "no-var": "error",
    "prefer-const": "warn"
  }
}

รัน lint

terminal (ใน backend/)
npm run lint

แก้ error ที่พบบ่อย

ESLint Errorสาเหตุวิธีแก้
Missing semicolonไม่มี ; ท้ายบรรทัดเพิ่ม ;
no-unused-varsประกาศ variable แต่ไม่ใช้ลบหรือใช้งาน variable นั้น
eqeqeqใช้ == แทน ===เปลี่ยนเป็น ===
no-varใช้ var แทน let/constเปลี่ยนเป็น const หรือ let
💡
แก้ error อัตโนมัติ รัน npx eslint . --fix เพื่อให้ ESLint แก้ error บางตัว (เช่น semicolons, quotes) อัตโนมัติ
✅ Checkpoint
  • npm run lint รันได้โดยไม่มี error (warnings OK)
  • ไฟล์ backend/.eslintrc.json มีอยู่

สร้าง CI Workflow ที่สมบูรณ์ (3 Jobs)
⏱ 30 นาที

ถึงเวลาสร้าง CI workflow ที่สมบูรณ์ — 3 jobs ที่รันแบบ sequential: lint → test → build

Pipeline Flow

Push/PR
feature/**
→
Job: lint
ESLint
→
Job: test
Jest + Coverage
→
Job: build
Docker build

แก้ .github/workflows/ci.yml

.github/workflows/ci.yml (ฉบับสมบูรณ์)
name: CI

on:
  push:
    branches:
      - main
      - develop
      - 'feature/**'
      - 'hotfix/**'
    paths:
      - 'backend/**'
      - '.github/workflows/ci.yml'
  pull_request:
    branches:
      - main
      - develop
    types: [opened, synchronize, reopened]

jobs:
  # ─── JOB 1: LINT ───────────────────────────────────────────
  lint:
    name: 🔍 Lint
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: backend

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
          cache-dependency-path: backend/package-lock.json

      - name: Install dependencies
        run: npm ci

      - name: Run ESLint
        run: npm run lint

  # ─── JOB 2: TEST ───────────────────────────────────────────
  test:
    name: 🧪 Test & Coverage
    runs-on: ubuntu-latest
    needs: lint           # รอ lint ผ่านก่อน
    defaults:
      run:
        working-directory: backend

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
          cache-dependency-path: backend/package-lock.json

      - name: Install dependencies
        run: npm ci

      - name: Run tests with coverage
        run: npm test
        env:
          NODE_ENV: test

      - name: Upload coverage report
        uses: actions/upload-artifact@v4
        if: always()       # upload แม้ test fail เพื่อดู report
        with:
          name: coverage-report
          path: backend/coverage/
          retention-days: 7

  # ─── JOB 3: BUILD ──────────────────────────────────────────
  build:
    name: 🐳 Docker Build
    runs-on: ubuntu-latest
    needs: test           # รอ test ผ่านก่อน

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Build Docker image (no push)
        run: |
          docker build -t stockpro-backend:${{ github.sha }} ./backend
          echo "Build successful: stockpro-backend:${{ github.sha }}"

อธิบาย YAML ทีละส่วน

Keyความหมาย
defaults.run.working-directoryทำให้ทุก run: step เริ่มที่โฟลเดอร์ backend/ โดยอัตโนมัติ
actions/setup-node cache: 'npm'Cache node_modules → install เร็วขึ้น 60-80%
npm ciInstall ตาม package-lock.json แน่นอน ไม่อัปเดต (ปลอดภัยกว่า npm install)
needs: lintJob นี้จะรันเฉพาะเมื่อ lint ผ่าน ถ้า lint fail → test ไม่รัน
if: always()Upload artifact แม้ step ก่อนหน้าจะ fail
retention-days: 7เก็บ artifact ไว้ 7 วัน แล้วลบอัตโนมัติ
✅ Checkpoint
  • ci.yml มี 3 jobs: lint, test, build
  • test job มี needs: lint และ build job มี needs: test
  • มี upload-artifact step ใน test job

Branch Protection + Required CI Checks
⏱ 15 นาที

ตั้งค่า Branch Protection ให้ PR ถูก block จนกว่า CI จะผ่าน — ป้องกัน code ที่พังเข้า main/develop

ขั้นตอนตั้งค่าใน GitHub

  1. ไปที่ repo → Settings → Branches
  2. คลิก Add branch ruleset (หรือ Edit rule ที่มีอยู่)
  3. Branch name pattern: main
  4. เปิด Require status checks to pass
  5. ค้นหาและเพิ่ม: 🔍 Lint, 🧪 Test & Coverage
  6. เปิด Require branches to be up to date
  7. Save changes
ℹ️
Status check name = "name:" ใน workflow jobs ชื่อที่ต้องใส่ใน Required checks คือค่าจาก name: ของ job เช่น 🔍 Lint และ 🧪 Test & Coverage

ทดสอบ: เปิด PR แล้วดูว่า checks แสดงขึ้น

terminal
# ทำงานบน feature branch อยู่แล้ว
# เปิด PR ไป develop
gh pr create --title "feat: add unit tests and CI pipeline" \
  --body "Add Jest unit tests, ESLint config, and update CI workflow" \
  --base develop

ใน PR จะเห็น status checks รันอยู่ — ถ้า CI ผ่าน ปุ่ม Merge จะเปิดใช้ได้ ถ้า CI fail ปุ่ม Merge จะถูก block

✅ Checkpoint
  • Branch Protection Rule ของ main ต้องการให้ "Lint" และ "Test & Coverage" ผ่าน
  • เปิด PR แล้วเห็น CI checks รันใน PR page

Debug Challenge — แก้ Broken CI
⏱ 25 นาที

ครูจะ push commit ที่ทำให้ CI fail เข้าไปใน repo ของทีม ภารกิจของทีมคือ: อ่าน log, วิเคราะห์หาต้นเหตุ, แก้ไข, push ใหม่ให้ CI เขียวภายใน 20 นาที

🐛 Debug Protocol — ทำตามลำดับนี้ทุกครั้งที่ CI fail
1
อ่าน log จาก job ที่ fail — คลิก job สีแดง → คลิก step ที่ fail → อ่าน output สีแดงบรรทัดสุดท้าย
2
จำแนก error ประเภท — lint error? test fail? coverage ต่ำ? build fail? แต่ละประเภทแก้ต่างกัน
3
Reproduce ที่เครื่องตัวเอง — รัน npm run lint หรือ npm test บนเครื่อง แก้ให้ผ่านก่อน
4
แก้ code → commit → push — CI จะ trigger อีกรอบ รอดูว่าเขียวแล้ว
5
ถ้าแก้ไม่ได้ใน 5 นาที — ขอ hint จากครู อย่าเสียเวลากับ error เดิมนานเกิน

Error ที่พบบ่อยใน CI และวิธีแก้

Error ใน Logสาเหตุวิธีแก้
Cannot find module '../db'ลืม mock db หรือ path ผิดเพิ่ม jest.mock('../db', ...)
expect(received).toBe(expected)ค่าที่ได้ไม่ตรงกับที่คาดไว้ดูว่า API return อะไรจริง ๆ
Jest: "global" coverage threshold not metCoverage ต่ำกว่า thresholdเพิ่ม test cases ให้ครอบคลุม code มากขึ้น
error Parsing error: Unexpected tokenESLint parse JS ไม่ได้ (syntax error)แก้ syntax ใน JS file ที่ระบุ
npm ERR! missing script: lintไม่มี lint script ใน package.jsonเพิ่ม "lint": "eslint . --ext .js"
🚨
กฎ: ห้าม skip test หรือลด threshold เพื่อให้ CI ผ่าน การแก้ปัญหาที่ถูกต้องคือแก้ code ให้ test ผ่านจริง ไม่ใช่ลดมาตรฐานลง
✅ Checkpoint
  • วิเคราะห์ error จาก CI log ได้ว่า fail ที่ step ไหน เพราะอะไร
  • แก้ code จนผ่าน CI ได้ภายใน 20 นาที
  • PR page แสดง CI checks ทั้งหมดสีเขียว
