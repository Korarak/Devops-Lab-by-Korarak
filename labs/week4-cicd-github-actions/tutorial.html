⏱ 20 นาที

GitFlow เป็น branching model ที่กำหนดว่า branch ไหนทำหน้าที่อะไร ทำให้ทีมทำงานพร้อมกันโดยไม่ conflict กัน

── GitFlow Diagram ──────────────────────────────────

v1.0.0
develop
feature/stats
→ PR → develop

หน้าที่ของแต่ละ Branch

Branchหน้าที่Merge ไปที่
mainโค้ด production เท่านั้น — ห้าม push ตรง—
developรวม features ที่เสร็จแล้ว รอ releasemain
feature/*พัฒนา feature ใหม่ แยกจาก developdevelop
release/*เตรียม release (แก้ minor bug)main + develop
hotfix/*แก้ bug ด่วนใน productionmain + develop
💡
กฎทอง: ห้าม push ตรงเข้า main และ develop ทุก change ต้องผ่าน Pull Request เสมอ ให้เพื่อนร่วมทีม review ก่อน merge

สร้าง develop Branch + Branch Protection Rules
⏱ 20 นาที

สร้าง develop branch

bash 
git checkout -b develop
git push -u origin develop

ตั้งค่า Branch Protection — บน GitHub

Protection ป้องกันไม่ให้ push ตรงเข้า main ได้ โดยไม่ผ่าน PR

  1. repo → Settings → Branches
  2. Add branch protection rule
  3. Branch name pattern: main
  4. เปิดใช้:
    • ☑ Require a pull request before merging
    • ☑ Require approvals → 1
    • ☑ Require status checks to pass before merging
    • ☑ Do not allow bypassing the above settings
  5. คลิก Create
  6. ทำซ้ำสำหรับ develop (ผ่อนคลายกว่า ไม่ต้องใส่ require approval)
ℹ️
ทำไมต้อง Branch Protection? ถ้าไม่มี protection ใครก็ push เข้า main ตรงๆ ได้ ซึ่งอาจทำให้ production พัง Protection บังคับให้ทุก change ผ่าน review เสมอ
✅ ตรวจสอบ
  • ลอง git push origin main ตรงๆ → ต้องถูก block โดย GitHub
  • Settings → Branches → เห็น rule สำหรับ main แสดงอยู่

เพิ่ม /api/stats Endpoint ใน Backend
⏱ 30 นาที

แก้ไข backend/index.js — ทำ 3 อย่าง: (1) เพิ่ม const START_AT = Date.now() หลัง const PORT (2) แทนที่ /health เดิมด้วย version ใหม่ที่แสดง version+uptime (3) เพิ่ม /api/stats ใหม่ต่อจาก /health

🆕
ไฟล์ที่แก้: backend/index.js — เพิ่ม START_AT หลัง const PORT, แทนที่ /health เดิม, และเพิ่ม /api/stats ต่อจาก /health ก่อน routes /api/products
javascript · backend/index.js — เพิ่มส่วน health ใหม่ 
const START_AT = Date.now();   // เพิ่มบนสุด หลัง const PORT

// ── Health Check (ปรับปรุง: เพิ่ม version + uptime) ──────────
app.get('/health', async (req, res) => {
  try {
    const { rows } = await pool.query('SELECT NOW() AS db_time');
    res.json({
      status:    'ok',
      service:   'StockPro API',
      version:   process.env.APP_VERSION || '1.0.0',
      uptime_s:  Math.floor((Date.now() - START_AT) / 1000),
      db_time:   rows[0].db_time,
      timestamp: new Date(),
    });
  } catch (err) {
    res.status(503).json({ status: 'error', message: err.message });
  }
});

// ── GET /api/stats (endpoint ใหม่) ───────────────────────────
app.get('/api/stats', async (req, res) => {
  try {
    // Promise.all = รัน 2 queries พร้อมกัน ไม่ต้องรอทีละอัน
    const [overview, byCategory] = await Promise.all([
      pool.query(`
        SELECT
          COUNT(*)                                          AS total_products,
          SUM(stock)                                        AS total_stock,
          ROUND(SUM(price * stock), 2)                      AS total_value,
          COUNT(*) FILTER (WHERE stock = 0)                 AS out_of_stock,
          COUNT(*) FILTER (WHERE stock > 0 AND stock < 10)  AS low_stock
        FROM products
      `),
      pool.query(`
        SELECT
          category,
          COUNT(*)                     AS products,
          SUM(stock)                   AS total_stock,
          ROUND(SUM(price * stock), 2) AS total_value
        FROM products
        GROUP BY category
        ORDER BY total_value DESC
      `),
    ]);
    res.json({
      overview:    overview.rows[0],
      by_category: byCategory.rows,
    });
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

อธิบาย SQL ที่ใช้

  • COUNT(*) FILTER (WHERE stock = 0) — นับเฉพาะ rows ที่ stock = 0 (PostgreSQL syntax พิเศษ)
  • ROUND(SUM(price * stock), 2) — คำนวณมูลค่าสต็อกรวม ปัดทศนิยม 2 ตำแหน่ง
  • GROUP BY category ORDER BY total_value DESC — จัดกลุ่มตาม category เรียงตามมูลค่ามากไปน้อย

CI Workflow — อ่าน ci.yml ทีละบรรทัด
⏱ 35 นาที
🆕
ไฟล์ใหม่: .github/workflows/ci.yml

GitHub Actions อ่าน workflow file จากโฟลเดอร์ .github/workflows/ อัตโนมัติ ไฟล์ใช้ YAML format — indentation สำคัญมาก

yaml · .github/workflows/ci.yml (พร้อม comment อธิบาย) 
name: CI — StockPro    # ชื่อที่โชว์ใน GitHub Actions tab

# ── on: กำหนด "เมื่อไหร่" จะ trigger ─────────────────────────
on:
  push:
    branches:
      - develop          # ทริกเกอร์เมื่อ push ไป develop
      - 'feature/**'     # ทริกเกอร์เมื่อ push ไป feature/xxx
  pull_request:
    branches:
      - develop          # ทริกเกอร์เมื่อเปิด PR เข้า develop
      - main             # ทริกเกอร์เมื่อเปิด PR เข้า main

# ── jobs: งานที่ต้องทำ ─────────────────────────────────────────
jobs:
  test-and-build:
    name: Test & Build
    runs-on: ubuntu-latest    # ใช้ Runner Ubuntu ฟรีจาก GitHub

    steps:
      # ── Step 1: ดึง source code มาที่ Runner ─────────────────
      - name: Checkout code
        uses: actions/checkout@v4    # Action สำเร็จรูปจาก marketplace

      # ── Step 2: ติดตั้ง Node.js 20 ───────────────────────────
      - name: Setup Node.js 20
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      # ── Step 3-4: Backend ─────────────────────────────────────
      - name: Install backend dependencies
        working-directory: backend    # cd เข้าไปใน folder backend/ ก่อนรัน
        run: npm ci                   # ci = clean install (เร็วกว่า npm install)

      - name: Lint backend code
        working-directory: backend
        run: npm run lint             # รัน ESLint ตรวจ code style

      # ── Step 5-6: Frontend ────────────────────────────────────
      - name: Install frontend dependencies
        working-directory: frontend
        run: npm ci

      - name: Build frontend (production mode)
        working-directory: frontend
        run: npm run build    # Vite build → สร้าง dist/ folder

      # ── Step 7: Docker smoke test ─────────────────────────────
      - name: Build Docker images (smoke test)
        run: docker compose build    # ทดสอบว่า Dockerfile ทั้งคู่ build ได้

CD Workflow — อ่าน cd.yml ทีละบรรทัด
⏱ 35 นาที
🆕
ไฟล์ใหม่: .github/workflows/cd.yml

CD workflow ทำงานเมื่อ merge เข้า main หรือ push tag v* โดย build Docker image → push Registry → SSH deploy

yaml · .github/workflows/cd.yml (พร้อม comment อธิบาย) 
name: CD — Deploy StockPro

on:
  push:
    branches:
      - main       # merge เข้า main → deploy production
    tags:
      - 'v*'       # push tag v1.0.0 → build versioned image

jobs:
  # ── Job 1: Build + Push Docker images ─────────────────────────
  build-and-push:
    name: Build & Push Images
    runs-on: ubuntu-latest
    outputs:
      version: ${{ steps.version.outputs.tag }}    # ส่งค่าออกให้ Job 2

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      # กำหนด image tag: ถ้าเป็น git tag → ใช้ tag นั้น, ไม่งั้น → "latest"
      - name: Determine image version tag
        id: version
        run: |
          if [[ "$GITHUB_REF" == refs/tags/* ]]; then
            echo "tag=${GITHUB_REF#refs/tags/}" >> $GITHUB_OUTPUT
          else
            echo "tag=latest" >> $GITHUB_OUTPUT
          fi
          # GITHUB_REF ตัวอย่าง: refs/tags/v1.0.0
          # ${GITHUB_REF#refs/tags/} = ตัด prefix → เหลือ v1.0.0

      # Login เข้า Private Registry ด้วย Secrets
      - name: Login to private registry
        uses: docker/login-action@v3
        with:
          registry: ${{ secrets.REGISTRY_URL }}
          username: ${{ secrets.REGISTRY_USER }}
          password: ${{ secrets.REGISTRY_PASS }}

      # Build Backend image → push 2 tags (version + latest)
      - name: Build & push backend image
        uses: docker/build-push-action@v5
        with:
          context: ./backend
          push: true
          tags: |
            ${{ secrets.REGISTRY_URL }}/stockpro-backend:${{ steps.version.outputs.tag }}
            ${{ secrets.REGISTRY_URL }}/stockpro-backend:latest

      # Build Frontend image → push
      - name: Build & push frontend image
        uses: docker/build-push-action@v5
        with:
          context: ./frontend
          push: true
          tags: |
            ${{ secrets.REGISTRY_URL }}/stockpro-frontend:${{ steps.version.outputs.tag }}
            ${{ secrets.REGISTRY_URL }}/stockpro-frontend:latest

  # ── Job 2: SSH Deploy (รอ Job 1 เสร็จก่อน) ───────────────────
  deploy:
    name: Deploy to Server
    needs: build-and-push    # dependency — รอ Job 1
    runs-on: ubuntu-latest

    steps:
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1    # Action SSH สำเร็จรูป
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SERVER_SSH_KEY }}
          script: |
            cd /opt/stockpro
            docker compose -f docker-compose.prod.yml pull    # ดึง image ใหม่
            docker compose -f docker-compose.prod.yml up -d --remove-orphans
            docker image prune -f    # ลบ image เก่า
            echo "Deployed ${{ needs.build-and-push.outputs.version }} at $(date)"

GitHub Secrets — เก็บ Credentials อย่างปลอดภัย
⏱ 20 นาที
🔐
GitHub Secrets คืออะไร? เป็นที่เก็บ sensitive data (passwords, tokens, SSH keys) ที่ workflow อ่านได้ด้วย ${{ secrets.NAME }} แต่ GitHub ซ่อนค่าใน logs ทั้งหมด ไม่มีทางดูค่าจริงได้หลังจาก save

Secrets ที่ต้องตั้งค่า

Secret Nameค่าที่ใส่ใช้ใน
REGISTRY_URLregistry.lab.local:5000docker login
REGISTRY_USERusername ของ registrydocker login
REGISTRY_PASSpassword ของ registrydocker login
SERVER_HOSTIP address ของ serverSSH connect
SERVER_USERSSH username เช่น ubuntuSSH connect
SERVER_SSH_KEYSSH private key ทั้งหมด (รวม header)SSH auth

วิธีเพิ่ม Secret

  1. repo → Settings → Secrets and variables → Actions
  2. คลิก New repository secret
  3. ใส่ Name และ Secret value
  4. คลิก Add secret

สร้าง SSH Key Pair (ถ้ายังไม่มี)

bash 
# สร้าง key pair ใหม่สำหรับ GitHub Actions
ssh-keygen -t ed25519 -C "github-actions-stockpro" -f ~/.ssh/stockpro_deploy

# Public key → copy ไปใส่บน server
cat ~/.ssh/stockpro_deploy.pub
# ใส่ในไฟล์ ~/.ssh/authorized_keys บน server

# Private key → copy ทั้งหมดใส่ใน Secret: SERVER_SSH_KEY
cat ~/.ssh/stockpro_deploy
⚠️
ห้าม commit private key ลงใน repo ถ้า key หลุดขึ้น GitHub ให้ revoke ทันทีและสร้างใหม่ Private key ควรอยู่ใน Secrets เท่านั้น
✅ ตรวจสอบ
  • Settings → Secrets → Actions → เห็น secrets ทั้ง 6 ตัวในรายการ
  • ค่าของทุก secret ซ่อนอยู่ (•••••) ไม่แสดง plaintext

Release Tag v1.0.0 — Semantic Versioning
⏱ 15 นาที

Semantic Versioning (SemVer) คือมาตรฐานเลข version รูปแบบ MAJOR.MINOR.PATCH

  • MAJOR — breaking change (API ไม่ compatible กับเวอร์ชั่นเก่า)
  • MINOR — feature ใหม่ แต่ยัง backward compatible
  • PATCH — แก้ bug เล็กน้อย ไม่เพิ่ม feature
bash 
# ต้องอยู่บน main และ pull ล่าสุดก่อน
git checkout main
git pull origin main

# สร้าง annotated tag (มี message อธิบาย)
git tag -a v1.0.0 -m "Release v1.0.0: initial production release with /api/stats endpoint"

# Push tag ไป GitHub (CD จะทริกเกอร์อีกครั้ง!)
git push origin v1.0.0

สร้าง GitHub Release

  1. GitHub → Releases → Draft a new release
  2. Choose tag: v1.0.0
  3. Title: v1.0.0 — StockPro Initial Production Release
  4. เขียน Release Notes สิ่งที่มีใน version นี้
  5. คลิก Publish release
💡
CD ทริกเกอร์ซ้ำ! เมื่อ push tag v1.0.0 CD จะ build image อีกครั้ง และ tag image ด้วย v1.0.0 ด้วย ทำให้สามารถ rollback ไปเวอร์ชั่นเก่าได้ง่าย: image: registry/stockpro-backend:v1.0.0
✅ ตรวจสอบ
  • GitHub → Releases แสดง v1.0.0 พร้อม release notes
  • CD pipeline ทำงานอีกครั้งหลัง push tag
  • Registry มี image tag v1.0.0 แยกจาก latest
