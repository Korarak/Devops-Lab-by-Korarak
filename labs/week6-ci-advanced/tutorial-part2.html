useTasks.js — Key Patterns สำคัญ
Composable

เปิดไฟล์ frontend/src/composables/useTasks.js — ทบทวน patterns สำคัญที่ใช้ในโปรเจกต์

State ทั้งหมด:

tasksref([]) — array ของงานทั้งหมดที่ fetch จาก backend
loadingref(false) — แสดง spinner ระหว่าง API call
searchref('') — ค่า search input (debounced 280ms)
statusFilterref('') — ค่า status dropdown ('', 'todo', 'in-progress', 'done')
priorityFilterref('') — ค่า priority dropdown ('', 'low', 'medium', 'high')
showModalref(false) — toggle TaskModal visibility
editingIdref(null) — id ของงานที่กำลัง edit (null = add mode)
confirmDeleteref(null) — task object ที่รอ confirm ลบ
formreactive({}) — ข้อมูลใน TaskModal form (title, status, priority, ...)

Computed Properties:

useTasks.js — computed: filtered, stats
// filtered — tasks ที่ผ่าน 3 conditions (client-side filter)
const filtered = computed(() =>
  tasks.value.filter(t => {
    const matchSearch = !search.value ||
      t.title.toLowerCase().includes(search.value.toLowerCase()) ||
      (t.description || '').toLowerCase().includes(search.value.toLowerCase())
    const matchStatus   = !statusFilter.value   || t.status   === statusFilter.value
    const matchPriority = !priorityFilter.value || t.priority === priorityFilter.value
    return matchSearch && matchStatus && matchPriority
  })
)

// stats — คำนวณจาก tasks.value (ไม่ใช่ filtered)
const stats = computed(() => ({
  total:       tasks.value.length,
  todo:        tasks.value.filter(t => t.status === 'todo').length,
  in_progress: tasks.value.filter(t => t.status === 'in-progress').length,
  done:        tasks.value.filter(t => t.status === 'done').length,
  overdue:     tasks.value.filter(t =>
    t.due_date && new Date(t.due_date) < new Date() && t.status !== 'done'
  ).length
}))

Watch patterns — debounce + immediate:

useTasks.js — watchers
// search: debounce 280ms — ไม่ call API ทุก keystroke
let searchTimer = null
watch(search, () => {
  clearTimeout(searchTimer)
  searchTimer = setTimeout(fetchTasks, 280)
})

// statusFilter + priorityFilter: immediate fetch เมื่อเปลี่ยนค่า
watch([statusFilter, priorityFilter], fetchTasks)
debounce 280msรอ 280ms หลัง user หยุดพิมพ์แล้วค่อย fetch — ลด API calls ที่ไม่จำเป็น
watch([statusFilter, priorityFilter])watch หลาย ref พร้อมกัน — fetch ทันทีเมื่อ dropdown เปลี่ยน (ไม่ต้อง debounce เพราะ user click ไม่ได้ type)
🔄
Composable Pattern — เหตุผลที่สำคัญ Logic ใน useTasks.js สามารถ reuse ได้ในทุก component เพียง import ครั้งเดียว
ถ้าใช้ Vuex/Pinia แทน — ต้อง setup store, action, mutation แยก — สำหรับแอปขนาดนี้ composable ง่ายกว่ามาก
ถ้าใช้ Option API — logic จะกระจายใน data(), methods, computed, watch — อ่านยากกว่า

ci.yml — Jobs 1 & 2: Lint + Test
GitHub Actions

เปิดไฟล์ .github/workflows/ci.yml — ดู Jobs 1 และ 2 ที่เหมือนกับ Week 5 แต่เป็นส่วนฐานของ 4-job pipeline

ESLint config — .eslintrc.json:

ไฟล์: backend/.eslintrc.json
.eslintrc.json — ESLint configuration
{
  "env": {
    "node": true,
    "jest": true,
    "es2021": true
  },
  "extends": ["eslint:recommended"],
  "parserOptions": {
    "ecmaVersion": 2021
  },
  "rules": {

env.node: trueเปิด globals ของ Node.js เช่น require, module, process, __dirname
env.jest: trueเปิด globals ของ Jest เช่น describe, it, expect, beforeAll — ป้องกัน "no-undef" error ในไฟล์ test
eslint:recommendedrule set พื้นฐานที่ ESLint แนะนำ — ตรวจ common mistakes เช่น no-undef, no-dupe-keys
no-unused-vars: warnwarn (ไม่ error) เมื่อมีตัวแปรที่ไม่ได้ใช้ — ไม่ทำให้ pipeline fail แต่บอกให้รู้
no-console: offปิด rule นี้ — backend มี console.log/error เยอะ ถ้าเปิดจะ warn ทุกบรรทัด

paths: backend/**รัน CI เฉพาะเมื่อไฟล์ใน backend/ เปลี่ยน — ประหยัด CI minutes ถ้าแก้ไขเฉพาะ docs หรือ README

Job 2 — Test:

ci.yml — Job 2: Test
  test:
    name: Test
    runs-on: ubuntu-latest
    needs: lint

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4

      - name: Run Jest tests
        run: npm test
        working-directory: backend
        env:
          NODE_ENV: test

          path: backend/coverage/

needs: lintJob 2 จะ start เฉพาะเมื่อ Job 1 (lint) ผ่าน — ถ้า lint fail → test ไม่รัน → ประหยัด runner minutes
npm testรัน jest --coverage จาก scripts ใน package.json — สร้าง coverage/ folder พร้อม lcov.info
NODE_ENV: testset environment variable ให้ app รู้ว่าอยู่ใน test mode — index.js อ่านค่านี้เพื่อปรับ behavior
upload-artifact@v4save coverage/ folder เป็น artifact ใน GitHub Actions — ดาวน์โหลด HTML report ได้จาก Actions tab
if: always()upload artifact แม้ test fail — เพราะต้องการดูว่า test ไหนที่ fail โดยดูจาก coverage report
retention-days: 7เก็บ artifact ไว้ 7 วัน แล้วลบอัตโนมัติ — ป้องกัน storage ของ GitHub เต็ม

Job 3 (Sonar) ต้องรัน test ใหม่เพราะ SonarSource action ต้องการ lcov.info ในไฟล์ระบบของ runner นั้นเอง — ไม่สามารถใช้ artifact จาก Job อื่นได้โดยตรง

Job 3: SonarQube Scan — หัวใจของ Part 2
SonarQube

Job 3 คือส่วนใหม่ที่สำคัญที่สุดใน Week 6 — ทำ static analysis และตรวจ Quality Gate ก่อนที่ pipeline จะ proceed ไป Job 4 (Docker push)

SonarQube Architecture — ภาพรวมการทำงาน:

GitHub Actions Runner (ubuntu-latest) ├── npm test → สร้าง coverage/lcov.info └── sonarqube-scan-action@v3 ├── อ่านไฟล์: backend/index.js, tests/, coverage/lcov.info ├── อ่านค่า: backend/sonar-project.properties └── ส่งข้อมูลไปยัง: SONAR_HOST_URL (SonarQube server) using: SONAR_TOKEN (authentication) SonarQube Server (sonarcloud.io หรือ self-hosted) ├── analyze: code smells, bugs, vulnerabilities, duplications ├── import coverage จาก lcov.info → แสดง coverage % ใน dashboard └── evaluate Quality Gate → PASSED หรือ FAILED sonarqube-quality-gate-action@v1 └── poll ทุก 5s → รับผล Quality Gate ├── PASSED → Job 3 success → Job 4 เริ่มทำงาน └── FAILED → Job 3 fail → pipeline หยุด, Job 4 ไม่รัน

Job 3 YAML — เนื้อหาทั้งหมด:

ไฟล์: .github/workflows/ci.yml
ci.yml — Job 3: SonarQube Quality Gate
  sonar:
    name: SonarQube Quality Gate
    runs-on: ubuntu-latest
    needs: test

    steps:
      - name: Checkout code
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
          cache-dependency-path: backend/package-lock.json

      - name: Install dependencies
        run: npm ci
        working-directory: backend

      - name: Run tests to generate lcov coverage
        run: npm test
        working-directory: backend
        env:
          NODE_ENV: test

      - name: SonarQube Scan
        uses: SonarSource/sonarqube-scan-action@v3
        with:
          projectBaseDir: backend
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
          SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}

      - name: SonarQube Quality Gate check
        uses: SonarSource/sonarqube-quality-gate-action@v1
        timeout-minutes: 5
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
          SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}

อธิบายทุก field สำคัญ:

needs: testJob 3 รันเฉพาะเมื่อ Job 2 (test) ผ่าน — ถ้า 15 tests fail → Sonar ไม่รัน → ไม่เสีย Sonar API quota
fetch-depth: 0checkout git history ทั้งหมด (ไม่ใช่แค่ commit ล่าสุด) — SonarQube ใช้ git blame ระบุว่าใครเขียน code บรรทัดไหน เพื่อ assign ปัญหาให้ถูกคน
Run tests to generate lcovรัน Jest ใหม่อีกครั้งใน Job นี้ — lcov.info ต้องอยู่ใน filesystem ของ runner เดียวกับที่ Sonar scan
sonarqube-scan-action@v3official action จาก SonarSource — อ่าน sonar-project.properties แล้วส่งข้อมูลไปยัง SonarQube server
projectBaseDir: backendบอก action ว่า sonar-project.properties อยู่ใน backend/ ไม่ใช่ root — สำคัญมาก ถ้าไม่ใส่จะ error
SONAR_TOKENPersonal Access Token จาก SonarQube Settings → Security → Tokens — ใช้ authenticate กับ server (เก็บใน GitHub Secrets)
SONAR_HOST_URLURL ของ SonarQube server เช่น https://sonarcloud.io หรือ http://your-server:9000 (เก็บใน GitHub Secrets)
sonarqube-quality-gate-action@v1official action ที่ poll SonarQube server จนได้ผล Quality Gate — fail pipeline ถ้า gate ไม่ผ่าน
timeout-minutes: 5รอผล Quality Gate ไม่เกิน 5 นาที — ถ้า SonarQube server ช้ามาก/ไม่ตอบ pipeline จะ fail แทน hang ตลอดไป
⚠️
ลำดับ steps ใน Job 3 สำคัญมาก ต้องรัน npm test ก่อน sonarqube-scan-action เสมอ
เพราะ sonar-project.properties ระบุ sonar.javascript.lcov.reportPaths=coverage/lcov.info — ถ้าไฟล์นี้ไม่มีตอน scan → SonarQube จะแสดง coverage = 0% → Quality Gate fail
🔬
SonarCloud vs Self-hosted SonarQube SonarCloud (sonarcloud.io) — free สำหรับ public repo, ไม่ต้องตั้ง server, SONAR_HOST_URL=https://sonarcloud.io
Self-hosted SonarQube — ต้องตั้ง server เอง (Docker: sonarqube:community), SONAR_HOST_URL=http://your-ip:9000
สำหรับ course นี้: แนะนำ SonarCloud เพราะไม่ต้องตั้ง server

Quality Gate Concepts + ตั้งค่า GitHub Secrets
Secrets

ทำความเข้าใจ Quality Gate และตั้งค่า Secrets ที่จำเป็นใน GitHub repository

Quality Gate คืออะไร?

Quality Gate คือ ชุดเงื่อนไข ที่กำหนดใน SonarQube ว่า code ต้องผ่าน threshold เท่าไหร่จึงจะ "ผ่าน" — ถ้าไม่ผ่านข้อใดข้อหนึ่ง pipeline fail ทันที

Quality Gate PASSED
  • ✓ Coverage on New Code ≥ 80%
  • ✓ No new Critical Bugs
  • ✓ No new Vulnerabilities
  • ✓ No new Blockers
  • ✓ Duplication < 3%
Quality Gate FAILED
  • ✗ Coverage < threshold ที่กำหนด
  • ✗ มี new Critical Bug
  • ✗ มี new Vulnerability (HIGH/CRITICAL)
  • ✗ Duplication เกิน threshold
🎓
ปรับ Quality Gate สำหรับโปรเจกต์นักศึกษา Default coverage threshold คือ 80% ซึ่งอาจสูงเกินไปสำหรับโปรเจกต์เรียน
สามารถปรับใน SonarQube: Administration → Quality Gates → สร้าง gate ใหม่ แล้วตั้ง coverage ≥ 60%
ไป Settings → Projects → เลือก project → Quality Gate → เลือก gate ที่สร้าง

ขั้นตอนที่ 1 — สร้าง Project ใน SonarCloud:

1. ไปที่ sonarcloud.ioLogin ด้วย GitHub account
2. + → Analyze new projectเลือก GitHub organization/account → เลือก repository ของ TaskFlow
3. Set up projectเลือก "With GitHub Actions" → SonarCloud จะสร้าง project key และแนะนำ SONAR_TOKEN
4. Copy SONAR_TOKENบันทึก token ที่ขึ้นต้นด้วย sqp_xxxx... — แสดงครั้งเดียวเท่านั้น

ขั้นตอนที่ 2 — ตั้งค่า GitHub Secrets:

📍
วิธีเพิ่ม Secrets ใน GitHub ไปที่ Repository → Settings → Secrets and variables → Actions → New repository secret
GitHub Secrets ที่ต้องตั้งค่า
SONAR_TOKEN
Personal Access Token จาก SonarCloud — รูปแบบ: sqp_xxxxxxxxxxxxxxxxxxxx
สร้างได้ที่: SonarCloud → My Account → Security → Generate Tokens
SONAR_HOST_URL
URL ของ SonarQube server:
• SonarCloud: https://sonarcloud.io
• Self-hosted: http://your-server-ip:9000

ตรวจสอบว่า sonar.projectKey ตรงกัน:

⚠️
sonar.projectKey ต้องตรงกับ key ใน SonarCloud ในไฟล์ sonar-project.properties เราตั้งค่า sonar.projectKey=taskflow
ใน SonarCloud: project key ต้องเป็น taskflow ด้วย (หรือ format: orgname_taskflow)
ถ้าไม่ตรง → SonarQube จะ error "Project not found" หรือสร้าง project ใหม่แทนที่จะอัปเดต project เดิม

สิ่งที่เกิดขึ้นเมื่อ Quality Gate Fail:

Job 3 failsonarqube-quality-gate-action คืน exit code ไม่ใช่ 0 → step fail → Job 3 เป็น "Failed" ใน Actions UI
Job 4 ไม่รันJob 4 มี needs: sonar → เมื่อ Job 3 fail, Job 4 (docker push) ถูก skip อัตโนมัติ — ป้องกันการ push code ที่ไม่ quality ไป registry
PR blockedถ้า repo ตั้ง branch protection ให้ CI ต้องผ่าน → PR merge ไม่ได้จนกว่า Quality Gate จะผ่าน
