Quality Gate คืออะไร

Quality Gate คือชุดเงื่อนไข (conditions) ที่ SonarQube ใช้ตัดสินว่า code ผ่านมาตรฐานหรือไม่ ถ้า Gate = FAILED → Pipeline หยุดทันที ไม่ build image

SonarQube Architecture ในแวดวง CI/CD

diagram
GitHub Actions Runner
       │
       ├─ npm test --coverage  (สร้าง lcov.info)
       │
       ├─ sonar-scanner        (อ่าน sonar-project.properties + lcov.info)
       │    └── ส่งผลไปที่ SonarQube Server บน Proxmox
       │
       └─ quality-gate-check   (ดึงผล PASS/FAIL จาก SonarQube API)
            ├── PASSED → Pipeline ดำเนินต่อ
            └── FAILED → Pipeline หยุด ❌
💡
fetch-depth: 0 SonarQube ต้องการ git history เพื่อวิเคราะห์ New Code — ต้องใส่ fetch-depth: 0 ใน checkout action ไม่งั้น scanner จะไม่รู้ว่าโค้ดไหนใหม่ไหนเก่า

สร้างไฟล์ sonar-project.properties
⏱ 15 นาที

SonarQube Scanner อ่าน config จากไฟล์ sonar-project.properties ที่อยู่ใน project root (ใน folder backend/)

สร้างไฟล์ backend/sonar-project.properties

backend/sonar-project.properties
# SonarQube Project Configuration
sonar.projectKey=stockpro-team1
sonar.projectName=StockPro Team 1
sonar.projectVersion=1.0.0

# Source and test directories
sonar.sources=.
sonar.tests=tests
sonar.exclusions=node_modules/**,coverage/**,dist/**

# Coverage report path (generated by Jest --coverage)
sonar.javascript.lcov.reportPaths=coverage/lcov.info

# Language
sonar.language=js
sonar.sourceEncoding=UTF-8
ℹ️
sonar.javascript.lcov.reportPaths บอก SonarQube ว่า Coverage Report (LCOV format) อยู่ที่ไหน — Jest สร้างไฟล์นี้เมื่อรัน npm test ด้วย coverageReporters: ['lcov'] ใน jest.config.js

ตรวจสอบ jest.config.js รองรับ lcov แล้ว

backend/jest.config.js (ตรวจสอบ)
module.exports = {
  testEnvironment: 'node',
  testMatch: ['**/tests/**/*.test.js'],
  collectCoverageFrom: ['index.js', '!node_modules/**'],
  coverageThreshold: {
    global: { lines: 60, functions: 60, branches: 50 },
  },
  coverageReporters: ['text', 'lcov', 'html'],  // ← lcov ต้องมีอยู่แล้ว
};
✅ ตรวจสอบ
  • ไฟล์ backend/sonar-project.properties ถูกสร้างแล้ว
  • sonar.projectKey ตรงกับที่สร้างใน SonarQube
  • jest.config.js มี 'lcov' ใน coverageReporters
  • รัน npm test ใน backend/ แล้วมีไฟล์ coverage/lcov.info

เพิ่ม Job 3: SonarQube Scan ใน CI Workflow
⏱ 20 นาที

เปิดไฟล์ .github/workflows/ci.yml แล้วเพิ่ม Job ใหม่หลัง test job

.github/workflows/ci.yml — เพิ่ม sonar job
  # ─── JOB 3: SONARQUBE QUALITY GATE ──────────────────────────
  sonar:
    name: 🔬 SonarQube Quality Gate
    runs-on: ubuntu-latest
    needs: test

    steps:
      - name: Checkout code
        uses: actions/checkout@v4
        with:
          fetch-depth: 0        # ← สำคัญมาก: ต้องการ full history

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
          cache-dependency-path: backend/package-lock.json

      - name: Install dependencies
        run: npm ci
        working-directory: backend

      - name: Run tests to generate lcov coverage
        run: npm test
        working-directory: backend
        env:
          NODE_ENV: test

      - name: SonarQube Scan
        uses: SonarSource/sonarqube-scan-action@v3
        with:
          projectBaseDir: backend
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
          SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}

      - name: SonarQube Quality Gate check
        uses: SonarSource/sonarqube-quality-gate-action@v1
        timeout-minutes: 5
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
          SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}

อธิบายแต่ละ Step

  • fetch-depth: 0 — ดึง git history ทั้งหมด (ไม่ใช่แค่ commit ล่าสุด) เพื่อให้ SonarQube วิเคราะห์ New Code ได้แม่นยำ
  • npm test — รัน tests อีกครั้งเพื่อสร้าง coverage/lcov.info ในตัว sonar job
  • sonarqube-scan-action — รัน sonar-scanner บน projectBaseDir = backend
  • sonarqube-quality-gate-action — รอผล PASS/FAIL จาก SonarQube (timeout 5 นาที)
💡
ทำไมต้องรัน test ใน sonar job ด้วย? Artifact ข้ามระหว่าง Jobs ได้ แต่ actions/download-artifact ซับซ้อนกว่า — สำหรับ lab นี้รัน test ซ้ำใน sonar job ง่ายกว่าและชัดเจนกว่า
✅ ตรวจสอบ
  • ci.yml มี sonar job อยู่หลัง test job
  • sonar job ใช้ actions/checkout@v4 พร้อม fetch-depth: 0
  • sonarqube-scan-action ใช้ version @v3
  • SONAR_TOKEN และ SONAR_HOST_URL ใช้ ${{ secrets.XXX }}

ทำความเข้าใจ Quality Gate Check
⏱ 15 นาที

Step sonarqube-quality-gate-action จะ poll SonarQube API ทุก 10 วินาที จนกว่าการวิเคราะห์จะเสร็จ แล้วตรวจสอบผลลัพธ์

ผลลัพธ์ที่เป็นไปได้

Quality Gate Statusผลต่อ Pipelineสาเหตุ
PASSEDดำเนินต่อ → Job ถัดไปเป็นไปตาม Conditions ทั้งหมด
FAILEDPipeline หยุด ❌Coverage ต่ำเกินไป, มี Bug, มี Vulnerability
NONEข้ามการตรวจสอบProject ยังไม่มี Quality Gate กำหนด
TIMEOUTPipeline หยุด ❌SonarQube วิเคราะห์ไม่เสร็จใน 5 นาที

ทดสอบ: ทำให้ Quality Gate FAIL โดยเจตนา

⚠️
กิจกรรม: ทดสอบการ FAIL ลองลบ test case ออก 5 ตัว แล้ว push — Pipeline ควรหยุดที่ sonar job ถ้า coverage ตกต่ำกว่า 60% — ดู error message และคืน test case กลับมา
✅ ตรวจสอบ
  • เข้าใจว่า Quality Gate PASSED/FAILED ส่งผลต่อ Pipeline อย่างไร
  • เข้าใจว่า timeout-minutes: 5 หมายความว่าอะไร

ทำความรู้จัก docker/build-push-action
⏱ 15 นาที

docker/build-push-action@v5 เป็น Official Docker Action สำหรับ build และ push Docker image ใน GitHub Actions พร้อม BuildKit และ cache

เปรียบเทียบ Week 5 vs Week 6

Week 5 (smoke test)Week 6 (build-push)
คำสั่งdocker build ...docker/build-push-action@v5
Push Imageไม่ pushpush: true
BuildKitไม่ใช้setup-buildx-action
Layer Cacheไม่มีcache-from/to type=gha
Multi-tagไม่รองรับรองรับ (sha + latest)

ทำไมต้องใช้ setup-buildx-action ก่อน

อธิบาย
- name: Set up Docker Buildx
  uses: docker/setup-buildx-action@v3

# BuildX = Docker Buildkit mode
# ช่วยให้:
# - cache-from / cache-to ทำงานได้
# - Multi-platform build รองรับ
# - Build เร็วขึ้น 2-3x เพราะ layer cache
✅
GitHub Actions Cache (type=gha) cache-from: type=gha ใช้ GitHub Actions cache store เก็บ Docker layers — build ครั้งแรกช้า แต่ครั้งต่อๆ ไปเร็วมากเพราะ layer ที่ไม่เปลี่ยนจะถูก restore จาก cache

Image Tagging Strategy: SHA + Latest
⏱ 15 นาที

การตั้งชื่อ (Tag) Docker Image มีผลต่อความสามารถ rollback และ traceability ของระบบ

Tag Strategy ที่ใช้ใน Week 6

# git commit SHA git log --oneline -1 a3f7c91 feat: add product search filter # short SHA = 7 ตัวแรก git rev-parse --short HEAD a3f7c91 # Image Tags ที่จะ push registry.lab:5000/stockpro-backend:sha-a3f7c91 ← traceability registry.lab:5000/stockpro-backend:latest ← latest stable

ทำไมต้องมีทั้งสอง Tag

Tagประโยชน์ใช้เมื่อ
sha-a3f7c91รู้ว่า Image นี้มาจาก commit ไหน, rollback ได้แม่นยำเมื่อต้องการ trace ปัญหา
latestง่ายต่อการ pull, docker compose prod ใช้ latestdeploy ปกติ

วิธีดึง short SHA ใน GitHub Actions

yaml snippet
- name: Get short commit SHA
  id: vars
  run: echo "sha_short=$(git rev-parse --short HEAD)" >> $GITHUB_OUTPUT

# ใช้งาน:
# ${{ steps.vars.outputs.sha_short }}
💡
GITHUB_OUTPUT ใน GitHub Actions สมัยใหม่ (หลัง 2022) ใช้ echo "key=value" >> $GITHUB_OUTPUT แทน set-output เก่าที่ deprecated ไปแล้ว

เพิ่ม Job 4: Build & Push Docker Image
⏱ 25 นาที

เพิ่ม Job 4 ต่อจาก sonar job ใน ci.yml

.github/workflows/ci.yml — เพิ่ม build-push job
  # ─── JOB 4: BUILD & PUSH IMAGE ──────────────────────────────
  build-push:
    name: 🐳 Build & Push Image
    runs-on: ubuntu-latest
    needs: sonar
    if: github.ref == 'refs/heads/main' || github.ref == 'refs/heads/develop'

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Get short commit SHA
        id: vars
        run: echo "sha_short=$(git rev-parse --short HEAD)" >> $GITHUB_OUTPUT

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Login to private registry
        uses: docker/login-action@v3
        with:
          registry: ${{ secrets.REGISTRY_URL }}
          username: ${{ secrets.REGISTRY_USER }}
          password: ${{ secrets.REGISTRY_PASS }}

      - name: Build & push backend image
        uses: docker/build-push-action@v5
        with:
          context: ./backend
          push: true
          tags: |
            ${{ secrets.REGISTRY_URL }}/stockpro-backend:sha-${{ steps.vars.outputs.sha_short }}
            ${{ secrets.REGISTRY_URL }}/stockpro-backend:latest
          cache-from: type=gha
          cache-to: type=gha,mode=max

      - name: Build & push frontend image
        uses: docker/build-push-action@v5
        with:
          context: ./frontend
          push: true
          tags: |
            ${{ secrets.REGISTRY_URL }}/stockpro-frontend:sha-${{ steps.vars.outputs.sha_short }}
            ${{ secrets.REGISTRY_URL }}/stockpro-frontend:latest
          cache-from: type=gha
          cache-to: type=gha,mode=max

      - name: Summary
        run: |
          echo "## 🐳 Images Pushed" >> $GITHUB_STEP_SUMMARY
          echo "| Image | Tag |" >> $GITHUB_STEP_SUMMARY
          echo "|-------|-----|" >> $GITHUB_STEP_SUMMARY
          echo "| stockpro-backend | sha-${{ steps.vars.outputs.sha_short }} |" >> $GITHUB_STEP_SUMMARY
          echo "| stockpro-backend | latest |" >> $GITHUB_STEP_SUMMARY
          echo "| stockpro-frontend | sha-${{ steps.vars.outputs.sha_short }} |" >> $GITHUB_STEP_SUMMARY
          echo "| stockpro-frontend | latest |" >> $GITHUB_STEP_SUMMARY

อธิบาย Step Summary

Step Summary เขียนข้อมูลลงใน $GITHUB_STEP_SUMMARY — ผลจะปรากฏเป็น table สวยงามใน GitHub Actions Run Summary หน้า

✅ ตรวจสอบ
  • ci.yml มี 4 jobs ครบ: lint → test → sonar → build-push
  • build-push job มี needs: sonar
  • build-push ใช้ docker/build-push-action@v5
  • tags มีทั้ง sha-short และ latest
  • cache-from/cache-to type=gha ถูกตั้งค่าแล้ว

Conditional Step: รันเฉพาะ main/develop
⏱ 15 นาที

Job build-push ควรรันเฉพาะเมื่อ push ถึง main หรือ develop — ไม่ควร push image ทุกครั้งที่ push feature branch

วิธีใช้ if: condition

yaml — if condition
  build-push:
    name: 🐳 Build & Push Image
    runs-on: ubuntu-latest
    needs: sonar
    # ← รันเฉพาะ push ถึง main หรือ develop
    if: github.ref == 'refs/heads/main' || github.ref == 'refs/heads/develop'

ความหมายของ github.ref

กรณีgithub.refbuild-push รันไหม
push to feature/add-filterrefs/heads/feature/add-filter❌ ไม่รัน
push to developrefs/heads/develop✅ รัน
push to mainrefs/heads/main✅ รัน
PR opened (ไม่ merge)refs/pull/1/merge❌ ไม่รัน
ℹ️
ทำไม feature branch ไม่ควร push image Image ที่ push ขึ้น Registry คือ artifact ที่พร้อม deploy — ถ้า push ทุก feature branch จะมี image ขยะจำนวนมาก และอาจทำให้ deploy ผิด version
✅ ตรวจสอบ
  • build-push job มี if: condition ถูกต้อง
  • ทดสอบ: push ไป feature branch แล้ว build-push job ถูก skip (สีเหลือง)
  • ทดสอบ: push ไป develop แล้ว build-push job รันจริง

BuildKit Cache: เร่งความเร็ว Build
⏱ 15 นาที

Docker BuildKit cache ช่วยให้ build ครั้งที่ 2 เป็นต้นไปเร็วขึ้นมาก เพราะ layer ที่ไม่เปลี่ยนจะถูก cache ไว้

วิธีทำงานของ Layer Cache

Dockerfile + cache
FROM node:20-alpine           # Layer 1: cached (ไม่ค่อยเปลี่ยน)
WORKDIR /app                  # Layer 2: cached
COPY package*.json ./         # Layer 3: เปลี่ยนถ้า package.json เปลี่ยน
RUN npm ci --omit=dev         # Layer 4: ใช้ cache ถ้า Layer 3 ไม่เปลี่ยน (ประหยัดสุด!)
COPY . .                      # Layer 5: เปลี่ยนทุก push (code เปลี่ยน)
CMD ["node", "index.js"]      # Layer 6: cached

ผลการ Benchmark

สถานการณ์ไม่มี Cacheมี Cache (GHA)
เฉพาะ code เปลี่ยน (ไม่เปลี่ยน package.json)~2 นาที~20 วินาที
package.json เปลี่ยน (ต้อง npm ci ใหม่)~2 นาที~1.5 นาที
Build ครั้งแรกสุด~2 นาที~2 นาที
yaml — cache config อธิบาย
- name: Build & push backend image
  uses: docker/build-push-action@v5
  with:
    context: ./backend
    push: true
    tags: |
      ${{ secrets.REGISTRY_URL }}/stockpro-backend:sha-${{ steps.vars.outputs.sha_short }}
      ${{ secrets.REGISTRY_URL }}/stockpro-backend:latest
    cache-from: type=gha          # ← restore cache จาก GitHub Actions cache
    cache-to: type=gha,mode=max   # ← save ทุก layer กลับเข้า cache

Commit และ push workflow ใหม่ไปที่ develop branch แล้วตรวจดู Pipeline

ขั้นตอนที่ 14.1 — Commit และ Push

bash
git add .github/workflows/ci.yml backend/sonar-project.properties

ขั้นตอนที่ 14.2 — ติดตาม Pipeline

  1. ไปที่ GitHub Repository → Actions
  2. คลิก workflow run ล่าสุด
  3. ดู 4 Jobs: lint → test → sonar → build-push
  4. คลิก sonar job → ดู log ของ SonarQube Scan
  5. ไปที่ SonarQube Dashboard ดูผลการวิเคราะห์

ขั้นตอนที่ 14.3 — ตรวจสอบ Registry

หลัง build-push job สำเร็จ ตรวจสอบ image ใน Private Registry:

bash — ตรวจสอบ image ใน registry

docker login registry.lab:5000

# ดู list images
curl -k https://registry.lab:5000/v2/_catalog

# ดู tags ของ stockpro-backend
curl -k https://registry.lab:5000/v2/stockpro-backend/tags/list
