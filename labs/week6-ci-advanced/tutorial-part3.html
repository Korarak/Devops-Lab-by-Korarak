Job 4 Overview — ทำไม Conditional? ทำไมรอ Sonar?
Conditional Job
💡
Job 4 คือขั้นตอนสุดท้ายของ pipeline จะ build Docker image และ push ขึ้น registry — แต่ทำงานเฉพาะ main และ develop branch เท่านั้น และต้องผ่าน Job 1, 2, 3 ทุกตัวก่อน

ภาพรวม 4-Job Pipeline:

push to any branch (feature/*, hotfix/*)
🔍 Job 1: Lint ↓ needs: lint 🧪 Job 2: Test & Coverage ↓ needs: test 🔬 Job 3: SonarQube Quality Gate ↓ needs: sonar 🐳 Job 4: Build & Push ⛔ ไม่รัน (if condition false)
push to main / develop
🔍 Job 1: Lint ↓ needs: lint 🧪 Job 2: Test & Coverage ↓ needs: test 🔬 Job 3: SonarQube Quality Gate ↓ needs: sonar 🐳 Job 4: Build & Push ✅ รัน (if condition true)

Conditional ใน ci.yml — Job 4 มี 2 เงื่อนไข:

.github/workflows/ci.yml — Job 4 header
  build-push:
    name: 🐳 Build & Push Image
    runs-on: ubuntu-latest
    needs: sonar
    if: github.ref == 'refs/heads/main' || github.ref == 'refs/heads/develop'

อธิบายแต่ละส่วน:

needs: sonar Job 4 รอให้ Job 3 (sonar) ผ่านก่อน — และเพราะ sonar ต้องการ test ก็รอ test, test ต้องการ lint ก็รอ lint — ดังนั้น ต้องผ่านทั้ง 3 jobs
if: github.ref == 'refs/heads/main' ตรวจว่า branch ที่ push คือ main — ถ้าใช่ → รัน Job 4
|| github.ref == 'refs/heads/develop' หรือเป็น develop — branch staging ก็รัน Job 4 ด้วย
refs/heads/main รูปแบบเต็มของ branch reference ใน Git — ต้องใส่ refs/heads/ นำหน้าเสมอใน if condition
🎯
ทำไมต้อง Conditional? ประหยัดค่าใช้จ่าย: Feature branch มีการ push บ่อยมาก ถ้า build + push ทุกครั้งจะเปลือง registry storage และ CI minutes โดยไม่จำเป็น

ความปลอดภัย: image ใน registry ควรมาจาก branch ที่ผ่าน review แล้วเท่านั้น ไม่ใช่จาก experiment branch ของนักพัฒนา
⚠️
ถ้าโค้ดไม่ผ่าน Quality Gate → จะไม่มีการ build image เด็ดขาด เพราะ needs: sonar ทำให้ Job 4 รอ Job 3 — ถ้า Job 3 fail (Quality Gate ไม่ผ่าน) Job 4 จะถูก skip ไปอัตโนมัติ ระบบนี้รับประกันว่า image ทุกตัวที่อยู่ใน registry ผ่าน code quality check มาแล้ว

Docker Buildx + Login to Registry
Setup Steps

ก่อนจะ build และ push image ต้องตั้งค่า 3 ส่วนก่อน: รับ commit SHA, เปิดใช้ BuildKit, และ login เข้า registry

ส่วนนี้อยู่ใน: .github/workflows/ci.yml — Job 4 steps
ci.yml — Setup steps ใน Job 4
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Get short commit SHA
        id: vars
        run: echo "sha_short=$(git rev-parse --short HEAD)" >> $GITHUB_OUTPUT

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Login to private registry
        uses: docker/login-action@v3
        with:
          registry: ${{ secrets.REGISTRY_URL }}
          username: ${{ secrets.REGISTRY_USER }}
          password: ${{ secrets.REGISTRY_PASS }}

อธิบายทีละบรรทัด:

git rev-parse --short HEAD ดึง commit hash ปัจจุบันมาแบบสั้น 7 ตัวอักษร เช่น abc1234 — ใช้เป็น image tag เพื่อ traceability
>> $GITHUB_OUTPUT เขียนค่าลงใน special file ของ GitHub Actions เพื่อให้ step อื่น อ่านได้ผ่าน steps.vars.outputs.sha_short
id: vars ตั้งชื่อ step นี้ว่า vars — ทำให้ step ถัดไปอ้างถึงได้ว่า steps.vars.outputs.sha_short
setup-buildx-action@v3 เปิดใช้ Docker BuildKit — เป็น build engine รุ่นใหม่ที่รองรับ parallel layer build, advanced caching, และ multi-platform builds
login-action@v3 ทำ docker login โดยใช้ 3 secrets — credentials จะถูกเก็บใน runner memory ชั่วคราวสำหรับ session นี้เท่านั้น
registry: REGISTRY_URL endpoint ของ registry ที่จะ login — Docker Hub: docker.io, GitHub Container Registry: ghcr.io, หรือ IP ของ private registry
username / password credentials สำหรับ login — ควรใช้ access token แทน password ตรงๆ เพราะสามารถ revoke ได้โดยไม่ต้องเปลี่ยน password
🔐 Secrets ที่ต้องตั้งค่าใน GitHub Repository

ไปที่ GitHub repo → Settings → Secrets and variables → Actions → New repository secret

REGISTRY_URL Docker Hub: docker.io · GitHub Container Registry: ghcr.io · private registry: registry.example.com หรือ IP:port
REGISTRY_USER username สำหรับ login เข้า registry — สำหรับ Docker Hub คือ Docker Hub username ของคุณ
REGISTRY_PASS password หรือ access token — แนะนำ token (Docker Hub: Account Settings → Security → Access Tokens) เพราะ revoke ได้ไม่กระทบ account
💡
เหตุใด BuildKit ถึงเร็วกว่า Docker build ปกติ? Docker build แบบเดิมสร้าง layer ทีละชั้นตามลำดับ BuildKit วิเคราะห์ Dockerfile แล้วรัน step ที่ไม่พึ่งพากัน พร้อมกัน นอกจากนี้ยังรองรับ --mount=type=cache และ cache export/import ที่ใช้ใน step ถัดไป

build-push-action@v5 — อธิบายทุก Field
Docker BuildKit

docker/build-push-action@v5 คือ action หลักที่ build image และ push ขึ้น registry ในคำสั่งเดียว รองรับ BuildKit, cache, และ multi-tag อย่างสมบูรณ์

ส่วนนี้อยู่ใน: .github/workflows/ci.yml — Job 4
ci.yml — Build & Push backend + frontend
      - name: Build & push backend image
        uses: docker/build-push-action@v5
        with:
          context: ./backend
          push: true
          tags: |
            ${{ secrets.REGISTRY_URL }}/taskflow-backend:sha-${{ steps.vars.outputs.sha_short }}
            ${{ secrets.REGISTRY_URL }}/taskflow-backend:latest
          cache-from: type=gha
          cache-to: type=gha,mode=max

      - name: Build & push frontend image
        uses: docker/build-push-action@v5
        with:
          context: ./frontend
          push: true
          tags: |
            ${{ secrets.REGISTRY_URL }}/taskflow-frontend:sha-${{ steps.vars.outputs.sha_short }}
            ${{ secrets.REGISTRY_URL }}/taskflow-frontend:latest
          cache-from: type=gha
          cache-to: type=gha,mode=max

อธิบายทุก field อย่างละเอียด:

context: ./backend โฟลเดอร์ที่ส่งให้ Docker daemon เป็น build context — Docker จะส่งไฟล์ทั้งหมดในโฟลเดอร์นี้ไปยัง build daemon และหา Dockerfile ใน path นี้
push: true สั่ง push image ขึ้น registry หลัง build เสร็จ — ถ้าตั้ง push: false จะ build แต่ไม่ push (ใช้สำหรับทดสอบว่า Dockerfile ใช้ได้โดยไม่ต้องการ credentials)
tags: | (multi-line) ใช้ YAML literal block (|) เพื่อใส่หลาย tag — action จะ push image เดียวกัน แต่ tag ด้วยทุก tag ที่ระบุ โดย build แค่ครั้งเดียว
REGISTRY_URL/taskflow-backend:sha-abc1234 tag แบบ immutable — ผูกกับ commit hash เพื่อ traceability: รู้ว่า image นี้มาจาก commit ไหน ใช้สำหรับ rollback และ audit
REGISTRY_URL/taskflow-backend:latest tag แบบ mutable — ชี้ไปที่ image ล่าสุดที่ผ่าน CI เสมอ สะดวกสำหรับ docker pull ง่ายๆ และ CD pipeline ที่ต้องการ "เวอร์ชันล่าสุด"
cache-from: type=gha ดึง cached layers จาก GitHub Actions Cache — ถ้า run นี้มี cache จาก run ก่อน Docker จะ skip การ build layers ที่ไม่เปลี่ยน ทำให้เร็วขึ้นมาก
cache-to: type=gha,mode=max บันทึก layers ลง GitHub Actions Cache หลัง build เสร็จ เพื่อให้ run ถัดไปใช้ได้ — ต้องรัน cache-to อย่างน้อย 1 ครั้งก่อน cache-from ถึงจะได้ประโยชน์
mode=max vs mode=min mode=max cache ทุก layer รวมถึง intermediate layers → hit rate สูงกว่า แต่ใช้ storage มากกว่า | mode=min cache เฉพาะ final image layers → ประหยัด storage แต่ cache hit น้อยกว่า
⚡
ผลลัพธ์ที่เห็นได้จากการใช้ Cache Build ครั้งแรก (ไม่มี cache): ประมาณ 3–5 นาที
Build ครั้งถัดไป (มี cache, โค้ดเปลี่ยนนิดหน่อย): ประมาณ 30–60 วินาที
เพราะ layer ที่ไม่เปลี่ยน เช่น npm install จะถูก pull จาก cache แทนการรันใหม่
🏗️
ทำไมต้อง build ทั้ง backend และ frontend แยกกัน? แต่ละส่วนมี Dockerfile ของตัวเอง และ deploy บน container คนละตัว — backend เป็น Node.js API, frontend เป็น Nginx serve static files จาก Vue build การแยก image ทำให้ scale แต่ละส่วนได้อิสระ และ deploy บางส่วนโดยไม่ต้อง redeploy ทั้งระบบ

Image Tagging Strategy — sha vs latest
Tagging Strategy
🏷️
ทำไมต้องมี 2 tags? แต่ละ tag ตอบโจทย์คนละ use case — sha สำหรับ precision และ traceability, latest สำหรับ convenience การใช้แค่ tag เดียวจะเสียประโยชน์ของอีกแบบไป

เปรียบเทียบ 2 tagging strategy:

Tag ตัวอย่าง ใช้ทำอะไร ข้อดี ข้อเสีย
sha taskflow-backend:sha-abc1234 Traceable deploy, rollback ย้อนกลับได้แม่นยำ รู้ว่าเป็น commit ไหน tag สะสมเยอะขึ้นเรื่อยๆ ต้องมี cleanup policy
latest taskflow-backend:latest docker pull ล่าสุด, CD pipeline ง่าย ไม่ต้องจำ hash ใช้ได้ทันที ไม่รู้ว่าเป็น commit ไหน ถ้า image มีปัญหา rollback ยาก

ความหมายในเชิง immutability:

sha-abc1234 (immutable) "โค้ด commit abc1234 อยู่ใน image นี้แน่ๆ" — tag นี้จะไม่ถูก overwrite เพราะ sha ของแต่ละ commit ไม่ซ้ำกัน ใช้ใน production deploy และ audit trail
latest (mutable) "เป็น image ล่าสุดที่ผ่าน CI" — tag นี้ถูก overwrite ทุกครั้งที่มี build ใหม่ ใช้สำหรับ development/staging หรือ CD pipeline ที่ต้องการ pull ล่าสุดเสมอ
⚠️
Best Practice: อย่าใช้ :latest ใน Production! ใน production environment ควร deploy ด้วย sha tag เสมอ — เพราะถ้า deploy ด้วย :latest แล้วมีปัญหา คุณจะไม่รู้ว่า latest ณ เวลานั้นคือ commit ไหน และ rollback ไปเวอร์ชันที่ stable ทำได้ยาก

ตัวอย่าง Rollback ด้วย sha tag:

PowerShell — Rollback workflow
# ดูว่า container ไหนกำลังรันอยู่และใช้ image อะไร
docker ps --format "table {{.Names}}\t{{.Image}}\t{{.Status}}"

# ตัวอย่าง output:
# NAMES               IMAGE                                          STATUS
# taskflow-backend    registry.example.com/taskflow-backend:sha-abc1234   Up 2 hours

# Rollback ไปเวอร์ชันก่อนหน้า (สมมติ sha ก่อนหน้าคือ def5678)
docker pull registry.example.com/taskflow-backend:sha-def5678

docker stop taskflow-backend
docker rm taskflow-backend

docker run -d \
  --name taskflow-backend \
  --restart unless-stopped \
  -p 3000:3000 \
  registry.example.com/taskflow-backend:sha-def5678
✅
ทำไม sha tag ถึง rollback ได้แม่นยำกว่า? เพราะ sha-def5678 คือ image ที่ build มาจาก commit def5678 พอดี — คุณรู้แน่ว่าโค้ดนั้นเป็นอะไร ต่างจาก latest ที่อาจชี้ไป image คนละตัวกับตอนที่คุณ deploy

GitHub Actions Summary + Debug Challenges + Final Checklist
Wrap-up

Summary Step — แสดงผลใน GitHub Actions UI:

ci.yml — Summary step ท้าย Job 4
      - name: Summary
        run: |
          echo "## 🐳 Images Pushed" >> $GITHUB_STEP_SUMMARY
          echo "| Image | Tag |" >> $GITHUB_STEP_SUMMARY
          echo "|-------|-----|" >> $GITHUB_STEP_SUMMARY
          echo "| taskflow-backend | sha-${{ steps.vars.outputs.sha_short }} |" >> $GITHUB_STEP_SUMMARY
          echo "| taskflow-backend | latest |" >> $GITHUB_STEP_SUMMARY
          echo "| taskflow-frontend | sha-${{ steps.vars.outputs.sha_short }} |" >> $GITHUB_STEP_SUMMARY
          echo "| taskflow-frontend | latest |" >> $GITHUB_STEP_SUMMARY
$GITHUB_STEP_SUMMARY special environment variable ที่ชี้ไปยังไฟล์ Markdown ชั่วคราว — ทุกอย่างที่ echo ลงไปจะถูก render เป็น Markdown table ใน GitHub Actions run page (ไปดูได้ที่ Actions tab → เลือก run → Summary)
>> $GITHUB_STEP_SUMMARY ใช้ >> (append) ไม่ใช่ > (overwrite) เพราะต้องเพิ่มหลายบรรทัดต่อเนื่องกัน ถ้าใช้ > จะลบของเดิมทุกครั้ง

❓ Debug Challenges — ลองคิดก่อนดูเฉลย

สถานการณ์จริงที่เกิดขึ้นบ่อยใน CI/CD — ลองวิเคราะห์ก่อนดู hint

  • 1. Job 4 ไม่รัน ทั้งที่ push ไป main branch แล้ว?
    ตรวจสอบ if condition ใน ci.yml — ต้องเป็น refs/heads/main (มี refs/heads/ นำหน้า) ไม่ใช่แค่ main เปล่าๆ และตรวจว่า Job 3 (sonar) ผ่านแล้วหรือยัง เพราะ Job 4 ใช้ needs: sonar
  • 2. Build สำเร็จ แต่ push ล้มเหลวด้วย error "unauthorized: authentication required"?
    ตรวจสอบ REGISTRY_USER และ REGISTRY_PASS ใน GitHub Secrets ว่าถูกต้อง — สังเกตว่า secret name ตรงกับที่ใช้ใน ci.yml หรือไม่ (case-sensitive) และ login step ทำงานสำเร็จหรือเปล่า (ดูใน Actions log)
  • 3. cache-from ไม่ work — build ช้าเท่าเดิมทุก run ราวกับไม่มี cache?
    cache-from จะดึง cache ได้ก็ต่อเมื่อมีการ save cache ไว้แล้ว — ต้องให้ cache-to ทำงานสำเร็จอย่างน้อย 1 ครั้งก่อน นอกจากนี้ cache key ขึ้นกับ runner OS และ repository ถ้าเปลี่ยน runner จะไม่ hit cache เดิม
  • 4. SonarQube แสดง coverage 0% ทั้งที่ test ผ่านใน Job 2 แล้ว?
    Job 3 (sonar) ต้อง รัน tests ใหม่อีกครั้ง ในตัวมันเองเพื่อสร้าง lcov.info file — ตรวจว่า step Run tests to generate lcov coverage อยู่ก่อน SonarQube Scan และไฟล์ sonar-project.properties ชี้ path coverage ถูกต้อง
✅
Final Checklist — ครบ Week 6 CI Advanced

  • Registry มี image: taskflow-backend:sha-xxxx และ taskflow-backend:latest
  • Registry มี image: taskflow-frontend:sha-xxxx และ taskflow-frontend:latest
